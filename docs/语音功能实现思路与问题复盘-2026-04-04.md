# 语音功能实现思路与问题复盘（2026-04-04）

> 说明：本文档来自其他项目，当前仅作为本仓库语音模块实现的参考材料。文中提到的目录结构、文件路径、接口契约与本项目并不一一对应，后续实现时请以当前仓库的实际代码组织为准，重点复用其中的实现思路、边界处理和问题复盘。

## 1. 目标与边界

- 目标：让学习场景输入框支持语音输入，流程为「录音 -> 转写 -> 回填文本」。
- 协议：统一使用 `POST /speech/transcribe`（`multipart/form-data`），契约见 `docs/contracts/v0.9-speech-transcribe.yaml`。
- 边界：仅用于学习相关输入框，不用于登录、注册、Deck/Group 创建等非学习表单。

## 2. 端到端链路

1. 前端点击语音按钮，`useSpeechRecorder` 启动 `MediaRecorder` 收集音频分片。
2. 再次点击后停止录音，得到 `Blob`。
3. `speechApi` 将 `Blob` 转为 `File`，拼装 `file/language/scene` 上传。
4. 后端 `backend-hono` 的 `/speech/transcribe` 做鉴权、表单校验、大小校验、场景解析。
5. 后端通过 speech adapter 调用 provider `/audio/transcriptions`，返回转写文本。
6. 前端收到结果后按当前光标位置回填到 textarea。

## 4. 前端实现要点

### 4.1 录音层（`useSpeechRecorder`）

- 状态机：`idle | recording | stopping | error`，避免重复点击引发并发录音。
- MIME 选择：优先用户指定，回退到候选集合（`webm/mp4/ogg/mpeg`）。
- 时长保护：
  - 最长时长：`DEFAULT_MAX_DURATION_MS = 15000`，超时自动停止。
  - 最短时长：`DEFAULT_MIN_DURATION_MS = 500`，低于 0.5s 直接判定无效，不上传。
- 资源释放：停止后统一关闭 track、清理 timer、清空引用。

### 4.2 组件层（`SpeechEnabledTextarea`）

- 语音按钮即 toggle：
  - `recording` 时点击执行停止并触发转写。
  - 非录音态点击执行开始录音。
- 转写成功后插入策略：
  - 空输入框：整段填入，光标移到末尾。
  - 非空输入框：按当前选区插入，保持编辑连续性。
- 错误展示优先级：API 错误优先，其次录音器错误。

### 4.3 API 上传层（`speechApi`）

- `Blob -> File`：按 MIME 推导扩展名（如 `audio/mp4` -> `speech.m4a`）。
- 上传前守门：`file.size <= 0` 直接抛 `SpeechApiError(422)`，不发请求。
- 错误解析：兼容 `detail: string`、`message`、`detail[]` 三类后端错误体。

## 5. 后端实现要点

### 5.1 路由校验（`/speech/transcribe`）

- 必须登录（session）。
- 必须为 `multipart/form-data`。
- 必须有 `file` 且 `size > 0`。
- 必须有 `scene` 与 `language`。
- 文件大小受 `STT_MAX_FILE_SIZE_BYTES` 限制（超限 413）。

### 5.2 场景映射（`speech/runtime.ts`）

- scene：`record_source | record_generated | review_answer | card_front | card_back`
- 每个 scene 绑定默认语言与 prompt，避免前端拼 prompt。
- 环境变量统一解析：`STT_PROVIDER/STT_MODEL/STT_TIMEOUT_MS/STT_FEATURE_ENABLED` 等。

### 5.3 Provider 适配（`aihubmix-adapter.ts`）

- 封装第三方 `/audio/transcriptions` 调用，路由层不耦合第三方细节。
- 超时控制：`AbortController`。
- 错误转换：统一抛 `SpeechProviderUnavailableError(message, status)`。
- fetch 绑定：`globalThis.fetch.bind(globalThis)`，避免上下文丢失。

## 6. 已修复 Bug 与根因

### Bug A：录音多久都报 “Audio file is too short”

- 现象：前端停止录音后，后端调用 provider 返回 `Audio file is too short`，页面显示 500。
- 根因：停止录音前调用 `requestData()` 与 `stop()` 触发浏览器事件时序竞态，导致最终仅上传到很短尾包。
- 修复：
  - 移除停止前 `requestData()`。
  - 保留最短时长 0.5 秒守门。
  - 新增竞态回归测试，模拟 “requestData 数据晚到” 场景。

### Bug B：空音频请求进入后端

- 现象：极端情况下会发出空 payload 请求。
- 修复：前端 `speechApi` 增加 `file.size <= 0` 直接失败。

### Bug C：provider 返回状态码被吞掉

- 现象：上游如 `429` 被统一映射为 `503/500`，影响定位与重试策略。
- 修复：后端按 `SpeechProviderUnavailableError.status` 透传（400~599），并补测试覆盖。

## 7. 踩坑清单（下个项目直接复用）

1. 不要默认在 `stop()` 前调用 `requestData()`，先确认目标浏览器实现是否存在时序差异。
2. 必须做“双守门”：录音最短时长 + 上传前文件非空校验。
3. MIME 与文件后缀要匹配，否则 provider 侧可能误判或降级处理。
4. 错误码不要“一刀切 500/503”，429/422/413 都应保留语义。
5. scene 必须白名单化，避免非学习表单误接入语音入口。
6. 单测必须覆盖异步竞态，而不只覆盖 happy path。
