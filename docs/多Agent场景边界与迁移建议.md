# 多 Agent 场景边界与迁移建议

> 当前状态：讨论稿  
> 创建日期：2026-05-23  
> 关联文档：`docs/multi-agent-design.md`、`docs/multi-agent-implementation-plan.md`、`docs/项目升级路线图与测试审计.md`  
> 文档目的：记录当前项目中多 Agent 的合理使用场景、边界、反模式，以及后续从单 Agent 迁移到多 Agent 的判断标准。

## 1. 背景

当前项目存在两条 AI 路线：

- **单 Agent**：当前 Cloudflare 部署主线，用于先完成生产化能力。
- **多 Agent**：基于 LangGraph，当前属于后续目标和能力练习方向。

多 Agent 的引入不只是为了业务复杂度，也是为了展示和练习 LLM 工程能力：

- Supervisor 路由
- 多 Agent 协作
- 工具契约
- eval 对比
- tracing 与失败复盘
- 高风险动作确认

但多 Agent 不应该为了“看起来复杂”而强行拆分。合理目标是：

> 前期用单 Agent 建立生产级行为基线和工程化能力，后期在 Node.js 部署中用多 Agent 替代单 Agent。单 Agent 沉淀出的 eval、工具契约、会话状态、确认机制、观测体系应尽量迁移复用。

## 2. 总体判断

本项目可以使用多 Agent，但要控制边界。

合理的多 Agent 价值在于：

- 组合多个领域能力，例如任务、天气、提醒、日程。
- 让不同 Agent 持有清晰职责。
- 让复杂请求有可解释的协作链路。
- 通过 eval 证明多 Agent 相比单 Agent 在某些场景上有收益。

不合理的多 Agent 用法是：

- 把确定性规则拆成 Agent。
- 把简单 CRUD 拆成多个 Agent。
- 让 LLM 判断权限、时间冲突、重复任务这类必须稳定的规则。
- 为了作品展示牺牲真实使用体验。

核心原则：

> 确定性规则放在 service / tool / policy 层；模糊判断、跨领域协调、自然语言表达交给 Agent。

## 3. 场景一：创建任务时结合天气和提醒

### 3.1 用户场景

用户输入：

```text
周六上午去机场接人
```

或：

```text
明天下午去公园跑步，记得提醒我
```

这类请求同时涉及：

- 创建任务
- 判断日期和时间
- 判断是否与天气相关
- 查询当天或任务时段天气
- 根据天气风险调整提醒文案或提醒策略

### 3.2 推荐协作方式

```text
Supervisor
  -> Task Agent：解析任务意图，调用任务工具创建任务
  -> Weather Agent：仅在任务具备天气相关性时查询天气
  -> Notification Agent：根据任务时间和天气上下文安排提醒或生成提醒建议
```

### 3.3 不建议的拆法

不建议拆成：

```text
Task Agent
Conflict Agent
Duplicate Agent
Weather Agent
```

原因：

- 时间冲突检测是确定性业务规则。
- 重复任务检测应该稳定、可测试。
- 权限校验必须由 service 层执行。
- 把这些规则交给 LLM 会增加误判、延迟和 token 成本。

### 3.4 推荐边界

Task Agent 负责：

- 理解用户要创建什么任务。
- 提取 title / dueDate / startTime / endTime / timeSegment / priority / groupId 等参数。
- 调用任务工具。
- 将工具返回结果转成用户可读回复。

任务工具或 `TaskService` 负责：

- 权限校验。
- 时间字段合法性。
- 时间冲突检测。
- 重复任务检测。
- 数据库写入。
- 返回结构化 `ToolResult`。

Weather Agent 负责：

- 判断任务是否需要天气上下文。
- 查询天气。
- 返回可被其他 Agent 使用的天气摘要。

Notification Agent 负责：

- 根据任务时间、天气和用户意图生成提醒建议。
- 安排提醒记录。
- 生成提醒文案。

## 4. 场景二：任务到期前结合天气发起提醒

### 4.1 用户场景

系统在任务到期前触发提醒，例如：

```text
你今天下午 3 点要去机场接人。预计有雨，建议提前出门并带伞。
```

这个场景比“创建任务时顺手查天气”更自然地适合多 Agent，因为它涉及：

- 读取即将到期任务
- 判断任务是否需要天气上下文
- 查询天气
- 生成提醒内容
- 发送通知

### 4.2 推荐协作方式

```text
确定性调度器
  -> 读取即将到期任务
  -> 判断任务是否到达提醒触发时间
  -> 调用多 Agent 流程

多 Agent 流程
  -> Task Agent：提供任务上下文
  -> Weather Agent：查询当天或任务时段天气
  -> Notification Agent：生成提醒文案并发送或保存提醒
```

### 4.3 关键边界

提醒触发不应该由 Agent 决定。

应该由确定性代码负责：

- 哪些任务需要提醒。
- 何时触发提醒。
- 是否已经提醒过。
- 是否需要重试。
- 是否写入提醒日志。

Agent 可以负责：

- 判断提醒内容如何表达。
- 根据天气调整提醒文案。
- 在多个提醒策略之间做轻量选择。

原因：

- 提醒漏发是严重体验问题。
- 调度逻辑必须稳定。
- LLM 不适合承担定时器和幂等判断。

## 5. 推荐 Agent 边界

### 5.1 Supervisor Agent

职责：

- 判断用户请求涉及哪些领域。
- 分发到合适的子 Agent。
- 汇总结果。

不负责：

- 数据库写入。
- 权限判断。
- 时间冲突判断。
- 业务规则兜底。

### 5.2 Task Agent

职责：

- 任务创建、查询、完成、修改、删除的自然语言入口。
- 参数提取。
- 候选任务定位。
- 调用任务工具。

不负责：

- 绕过 `TaskService` 直接操作数据库。
- 自己判断用户是否有权限。
- 自己判断重复任务或时间冲突。

### 5.3 Weather Agent

职责：

- 查询天气。
- 给出结构化天气摘要。
- 判断天气对任务是否有影响。

适合触发的任务类型：

- 出行
- 接送
- 户外运动
- 露营
- 洗车
- 搬家
- 其他明显受天气影响的任务

不适合触发的任务类型：

- 写周报
- 交材料
- 线上会议
- 室内学习
- 无天气相关性的普通任务

### 5.4 Notification Agent

职责：

- 根据任务时间和上下文生成提醒策略。
- 写入提醒记录。
- 生成提醒文案。
- 在天气或日程风险存在时增强提醒内容。

不负责：

- cron / queue / 定时触发。
- 幂等判断。
- 判断提醒是否已经发送。

### 5.5 Calendar Agent

当前项目如果没有真实日历数据，Calendar Agent 的价值有限。

适合在以下条件满足后引入：

- 存在真实日程来源。
- 可以查询用户空闲时间。
- 可以和任务时间冲突检测形成互补。

在没有外部日历或真实 calendar 数据之前，不建议过度投资 Calendar Agent。

## 6. 不建议做成 Agent 的能力

以下能力应优先放在确定性代码中：

- 权限校验。
- 输入 schema 校验。
- tool 参数 runtime validation。
- 时间字段合法性。
- 时间冲突检测。
- 重复任务检测。
- 数据库事务。
- 幂等保护。
- 删除 / 修改确认状态。
- 提醒触发调度。
- 任务是否已经提醒过。

这些能力可以被 Agent 调用，但不应该由 Agent 自己实现。

## 7. 从单 Agent 迁移到多 Agent 的策略

### 7.1 单 Agent 阶段要沉淀的可迁移资产

单 Agent 生产化不是浪费。它应该沉淀：

- eval 基准集。
- 工具契约。
- `ToolResult` 结构。
- prompt 版本管理思路。
- session / confirmation state。
- token 预算与历史截断策略。
- 高风险动作确认机制。
- tracing / 日志字段。
- 失败样本库。

这些资产后续可以迁移到多 Agent。

### 7.2 不应过度投资的内容

不建议为单 Agent 特有 loop 做过度复杂抽象。

如果某个抽象只服务于当前单 Agent 的手写循环，且未来多 Agent 无法复用，应谨慎投入。

### 7.3 多 Agent 替代单 Agent 的前置条件

多 Agent 达到以下条件后，才考虑替代单 Agent：

1. 覆盖单 Agent 的核心 eval。
2. 高风险动作确认机制等价或更好。
3. API 返回契约保持兼容。
4. 延迟和 token 成本可接受。
5. Node.js 部署稳定。
6. tracing 能解释每个 Agent 的输入、输出、工具调用和失败原因。
7. 常见任务场景不比单 Agent 更不稳定。

### 7.4 单 Agent 退役条件

单 Agent 不应长期和多 Agent 平行维护。

推荐退役条件：

- 多 Agent 连续通过核心 eval。
- 多 Agent 覆盖创建、查询、完成、修改、删除、确认、拒绝等核心场景。
- 多 Agent 的失败率、误操作率、延迟和成本在可接受范围。
- 线上真实使用一段时间未出现不可接受问题。
- 文档和部署策略已经切换到 Node.js 多 Agent 主线。

满足这些条件后，单 Agent 可以进入 deprecated，随后移除。

## 8. Eval 对比建议

多 Agent 是否值得保留，不应只凭主观感觉。

建议用同一套 eval 比较单 Agent 与多 Agent：

| 指标 | 说明 |
| --- | --- |
| 意图识别准确率 | 是否正确判断创建、查询、完成、修改、删除、拒绝 |
| 工具调用准确率 | 是否调用正确 tool |
| 字段提取准确率 | 日期、时间、标题、优先级、群组等是否正确 |
| 误操作率 | 用户未要求时是否执行写操作 |
| 高风险动作安全率 | 修改 / 删除是否经过确认 |
| 冲突检出率 | 时间冲突、重复任务是否被检出 |
| 天气相关性判断 | 是否只在必要任务中查询天气 |
| 提醒质量 | 提醒时间和文案是否合理 |
| 平均响应时间 | 多 Agent 是否引入不可接受延迟 |
| 平均 token 成本 | 多 Agent 是否成本明显过高 |

只有当多 Agent 在关键场景中带来可解释收益，或者作为作品展示有明确价值时，复杂度才是可接受的。

## 9. 反模式清单

避免以下做法：

1. **为简单 CRUD 强行拆 Agent**
   - 例如拆成 CreateTaskAgent、DeleteTaskAgent、UpdateTaskAgent。

2. **让 LLM 执行确定性业务规则**
   - 例如让 Agent 判断权限、判断任务是否重复、判断是否有时间冲突。

3. **所有请求都走多 Agent**
   - 简单查询和简单创建可以走最短路径，不必总是 Supervisor 多轮分发。

4. **没有 eval 就迁移**
   - 多 Agent 看起来更复杂，但不等于效果更好。

5. **多 Agent 与单 Agent 长期双主线**
   - 双轨可以作为迁移路径，但不应无限期并存。

6. **Agent 共享隐式可变状态**
   - 关键状态应显式持久化，例如确认状态、会话状态、提醒状态。

## 10. 当前建议

短期：

- 继续完善 Cloudflare 单 Agent。
- 将 Agent 工程化能力做成可迁移资产。
- 不急于扩展现有多 Agent 半成品。

中期：

- 用 eval 固化单 Agent 行为基线。
- 设计多 Agent 的最小合理场景：
  - 创建天气相关任务。
  - 任务到期前结合天气发提醒。
  - 后续可能加入真实日历。

长期：

- Node.js 部署多 Agent。
- 用同一套 eval 对比单 Agent 与多 Agent。
- 多 Agent 达标后退役单 Agent。
