# 质量门禁总览

> 适用范围：TypeScript 全栈项目。  
> 本文档是质量门禁的顶层指南，具体配置细节见各专项文档。

## 什么是质量门禁

质量门禁是**代码合入主分支前的一组自动化检查**。它的职责不是保证代码"好"，而是拦截"明显不应该进入生产的变更"——编译不过、类型错误、未使用变量、测试失败、格式不一致等。

质量门禁不替代代码审查、手工验收和 Agent eval，但它是这些活动的前提：如果连编译都过不了，后面的审查和测试没有意义。

## 质量门禁在 AI 辅助开发中的特殊意义

AI（包括 Claude Code、Codex、Cursor 等）生成代码的速度远超人工，但这也带来了特有的风险：

1. **AI 倾向于生成"看起来对"的代码。** 它能通过自然语言推理写出结构合理的实现，但不会主动跑编译器。类型不匹配、导入缺失、接口变更后的下游断裂，这些只有 `tsc` 和 ESLint 能拦截。

2. **AI 修改一个文件时不感知全局影响。** 改了 shared 的类型定义，server 和 web 的消费方可能已经断了，但 AI 不会主动去跑 typecheck。质量门禁在 CI 里拦这一层。

3. **AI 容易引入隐性退化。** 比如删掉一个"看起来没用"的变量，实际是 side effect import；或者把 `let` 改成 `const` 导致循环逻辑变化。lint 和 test 能在提交前暴露这些问题。

4. **AI 生成的测试本身可能有类型错误。** 本项目就出现过这种情况：39 个存量类型错误全在 AI 生成的测试文件里（`FakeListChatModel` 的 `responses` 类型不匹配、`TaskSource` 枚举值写错），因为之前没有 typecheck 门禁，这些错误一直没被发现。

**结论：AI 越快，门禁越重要。** 没有门禁的 AI 开发等于在没有刹车的车上踩油门。门禁不是限制 AI 的效率，而是让你敢于让 AI 大步改动——因为改错了会被自动拦住。

## 门禁层次

| 层 | 检查什么 | 工具 | 专项文档 |
|---|---------|------|---------|
| **Lint** | 代码规范、未使用变量、反模式 | ESLint | [ESLint 质量门配置说明](./ESLint质量门配置说明.md) |
| **TypeCheck** | 类型安全、接口契约 | `tsc --noEmit` | [TypeCheck 质量门配置说明](./TypeCheck质量门配置说明.md) |
| **Test** | 业务逻辑正确性 | Vitest | — |
| **Build** | 构建产物能否正常生成 | Vite / Wrangler | — |
| **Format Check** | 代码格式一致性 | Prettier (`--check`) | — |
| **Dependency Audit** | 已知漏洞依赖 | `pnpm audit` | — |
| **Secret Scan** | 防止密钥/token 意外提交 | git hooks / CI action | — |

前四层是核心门禁，任何 TS 全栈项目都应该有。后三层按项目阶段和需求逐步加入。

## CI 中的质量门禁

本地门禁和 CI 门禁不是一回事：

- **本地门禁**：开发时自己运行，例如 `pnpm -C packages/server lint`。它能提前发现问题，但依赖开发者主动执行。
- **CI 门禁**：代码推到 GitHub 后自动运行。它不依赖记忆和习惯，是团队协作时更可靠的底线。

本项目当前使用 GitHub Actions 做 CI。可以把它理解成 GitHub 提供的一台临时机器：每次触发 workflow 时，它会拉取代码、安装依赖，然后按配置运行 lint、typecheck、test、build。

当前 `.github/workflows/deploy-cloudflare.yml` 的触发方式是：

| 触发方式 | 会做什么 |
| --- | --- |
| PR 合到 `main` 前 | 只运行质量检查，不部署生产环境 |
| push 到 `main` | 先运行质量检查，通过后部署 API 和 Web |
| 手动 `workflow_dispatch` | 先运行质量检查，通过后部署 API 和 Web |

当前 CI 的 `ci` job 会运行：

- `pnpm -C packages/shared lint`
- `pnpm -C packages/shared typecheck`
- `pnpm -C packages/server lint`
- `pnpm -C packages/server typecheck`
- `pnpm -C packages/server test:ci`
- `pnpm -C packages/web lint`
- `pnpm -C packages/web typecheck`
- `pnpm -C packages/web build`

只要其中任意一步失败，GitHub Actions 里的这次 workflow 就会变红。对 PR 来说，这代表“检查没过”；对 `main` 部署来说，这会阻止后续部署步骤继续执行。

需要注意：**workflow 变红不等于自动禁止合并。** 如果要让 GitHub 真正拦住不合格 PR，还需要在仓库的 Branch protection 或 Ruleset 中，把 `ci` job 设置为 required status check。这样 PR 没跑过质量门禁时，GitHub 才会禁止 merge。

## 在新项目或已有项目中配置门禁的步骤

### 已有项目

已有项目的核心矛盾是：**存量问题太多，全开会立刻阻塞开发**。策略是"先能跑通，再逐步收紧"。

1. **盘点现状**
   - 项目有没有 lint / typecheck / test 脚本？能不能跑通？
   - 如果跑不通，先记录错误数量和类型，不急着修。

2. **建立最小可用基线**
   - Lint：选择 `recommended`（不选 strict），格式规则交给 Prettier，冲突的关闭。存量问题多的规则先设为 `warn`，不设为 `error`。参考 [ESLint 质量门配置说明](./ESLint质量门配置说明.md)。
   - TypeCheck：确保每个包有 `typecheck` 脚本。严格度选项（`strict`、`noUnusedLocals` 等）在所有包统一。参考 [TypeCheck 质量门配置说明](./TypeCheck质量门配置说明.md)。
   - Test：确保已有测试能跑通。不稳定的测试先 skip 并记录，不要删。
   - Build：确保 `build` 脚本正常输出。

3. **修复存量问题使基线通过**
   - 按 lint → typecheck → test 的顺序修。lint 修最快（大多是删未使用导入、加 `_` 前缀），typecheck 次之，test 最慢。
   - 对于需要大量改动才能修的 lint 规则，可以暂时关闭或降为 `warn`，记录在文档中。

4. **接入 CI**
   - 至少在 PR 上跑 lint + typecheck + test + build。
   - 如果某一层暂时跑不通，可以先不阻塞合入（`continue-on-error`），但必须在 PR 页面可见。

5. **逐步收紧**
   - `warn` → `error`（当 warning 数量降到可控）。
   - 加入 format check（`prettier --check`）。
   - 加入 dependency audit。

### 新项目

新项目没有存量债务，可以一步到位。

1. **初始化时就配好 lint + typecheck + format**
   - Lint：`@eslint/js` recommended + `typescript-eslint` recommended + `eslint-config-prettier`。`no-explicit-any` 直接设 `error`。
   - TypeCheck：`strict: true` + `noEmit: true` + `noUnusedLocals` + `noUnusedParameters` + `noFallthroughCasesInSwitch`。
   - Format：Prettier，配 `.prettierrc`。

2. **写第一个测试再写业务代码**
   - 先跑通测试框架（Vitest），写一个 smoke test，确保 test 脚本可用。

3. **第一个 PR 前配好 CI**
   - lint + typecheck + test + build 四件事全部在 PR 上跑。从第一天就阻塞不合格代码。

4. **后续按需加入**
   - dependency audit、secret scan、commit lint 等按团队规模和项目敏感度决定。

## 给 AI 配置门禁时的提示模板

当你让 AI 为一个项目配置质量门禁时，可以用以下结构：

> 这是一个 [技术栈描述] 项目。请为它配置质量门禁：
>
> 1. ESLint：使用 v9 flat config，基线用 recommended（不用 strict），不启用 typed linting，格式交给 Prettier。[已有项目则补充：当前有 N 个 lint 错误，先把阻塞的规则降为 warn]
> 2. TypeCheck：确保每个包有 `typecheck` 脚本（`tsc --noEmit`）。严格度选项在所有包统一：`strict: true` + `noUnusedLocals` + `noUnusedParameters` + `noFallthroughCasesInSwitch`。
> 3. Test：确保测试框架可用，已有测试能跑通。
> 4. CI：在 PR 上跑 lint + typecheck + test + build，全部通过才能合入。
>
> 先跑一遍看有多少错误，报告结果后再决定如何处理。不要一次性修完所有问题。

## 门禁不做什么

- **不替代代码审查。** 门禁只能拦截机器可发现的问题。设计是否合理、命名是否清晰、抽象层次是否恰当，这些需要人判断。
- **不替代手工验收。** UI 体验、响应式布局、真实邮件是否送达，这些门禁管不了。
- **不追求 100% 覆盖率。** 覆盖率是指标不是目标。一个 80% 覆盖率但测试都在验证业务行为的项目，比 100% 覆盖率但全是测试实现细节的项目更安全。
