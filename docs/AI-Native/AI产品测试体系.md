# AI 产品测试体系

> 本文档梳理 AI 产品（特别是 AI Agent 产品）的测试体系，覆盖传统测试和 eval 测试两个层面。

## 一、AI 产品的测试全景

AI 产品的代码分两类：**确定性的**和**概率性的**。两类需要不同的测试手段，缺一不可。

```
AI 系统的测试 = 传统测试（确定性部分） + Eval（概率性部分）
                 ↑                          ↑
           工具逻辑、数据操作、           LLM 的理解、生成、
           规则引擎、格式处理             路由决策、回复质量
```

传统测试回答"代码对不对"，eval 回答"AI 表现好不好"。只有传统测试，LLM 出了问题你不知道；只有 eval，工具逻辑出 bug 你也不知道。

基础 agent loop 和 agent 编排的门槛已经明显下降——框架（LangGraph、CrewAI 等）把编排复杂度大幅降低了。但生产级 agent 的难点转移到了可靠性评估和持续改进：**"能工作"和"工作得好"之间的距离，比大多数人想的要大得多**。这个距离就是 eval + 数据飞轮要填的。Agent 编排是骨架，eval + 数据飞轮是血液——骨架搭起来产品能站住，但血液流动起来产品才能生长。

注意：Tracing（调用链路记录）是 eval 和失败样本收集的前置条件——没有 tracing，你连"AI 当时做了什么决策"都不知道，eval 归因和失败分析都无从做起。关于 tracing 的详细讨论见 [AI Native 开发指南 · 层面 3 可观测性](./AI-Native开发指南.md)。

## 二、传统测试在 AI 产品中的角色

AI 相关代码中有大量**确定性逻辑**，这些应该用传统测试覆盖，方式和普通代码完全一样。

传统测试分三层（测试金字塔）：

| 层级 | 测什么 | 特点 |
|------|--------|------|
| **单元测试** | 单个函数/模块的逻辑，mock 掉外部依赖 | 最快、最便宜、数量最多 |
| **集成测试** | 多个模块协作（service + 数据库、API 路由 + 中间件） | 不 mock 或少 mock，验证模块间契约 |
| **E2E 测试** | 从用户视角测完整流程（打开页面→操作→看到结果） | 最慢、最贵、数量最少 |

AI 代码中适合传统测试的部分：

| 代码 | 测试类型 | 为什么不用 eval |
|------|---------|---------------|
| 冲突检测（Dice 系数计算、时间区间比较） | 单元测试 | 纯算法，输入输出确定 |
| 工具参数校验（日期格式、必填字段） | 单元测试 | 规则逻辑，不涉及 LLM |
| Hallucination guard（意图关键词匹配） | 单元测试 | 规则逻辑 |
| Prompt 拼接（system prompt 包含正确的时间、用户信息） | 单元测试 | 字符串拼接，确定性 |
| 工具执行（create_task 正确写入数据库） | 集成测试 | 数据库操作，确定性 |
| SSE 流式输出格式 | 单元测试 | 协议格式，确定性 |

## 三、Eval 测试

### 是什么

Eval 是 AI 系统的质量度量体系——覆盖 AI 的所有行为决策，不仅是回复文本质量，还包括工具选择、参数提取、路由决策、指令遵循、安全合规等。传统测试是确定性的——同样的输入，期望同样的输出，通过就是通过。Eval 是概率性的——同样的 prompt，LLM 两次输出可能不同，所以不是判断"输出等不等于期望值"，而是判断"AI 的行为决策是否正确、结果是否达标"。

### 三个核心要素

**1. Eval 数据集（测什么）**

一组代表性的输入，覆盖你关心的场景。数据集需要均衡——只用失败样本会偏向 hard cases，不能代表真实分布。完整的数据集应包含：

- **典型成功路径**：高频请求中 AI 能正确处理的场景，用来确保改进不引入回归。
- **历史失败样本**：每个 AI 犯过的错都是一条 eval case，用来补强回归能力。
- **边界 case**：模糊意图、多义表达、极端输入。
- **对抗输入和安全 case**：prompt injection、越权请求。
- **线上分布抽样**：按真实用户请求分布采样，衡量整体产品质量。

来源：冷启动时手动构造关键场景（有 10 条真实样本比没有强 100 倍）；上线后从脱敏的用户请求中持续积累。

**2. 评分方式（怎么判）**

| 方式 | 做法 | 适合场景 |
|------|------|---------|
| **规则判分** | 正则、关键词检查、结构校验 | 输出格式确定的场景（工具调用参数、JSON 结构） |
| **LLM 判分** | 用另一个 LLM 当裁判，按 rubric 打分 | 开放式文本输出（回复质量、语气、准确性） |
| **人工判分** | 人看结果打分 | 高风险场景、LLM 判分不够可靠时 |

三种经常组合使用：能用规则判的先用规则（快且确定），规则判不了的用 LLM 判，关键场景再加人工抽检。

**LLM-as-judge 的校准约束：** LLM judge 本身也会出错，不能默认可靠。使用时需要：
- Rubric 必须清晰，尽量用 pass/fail 或结构化评分，减少模糊主观判断。
- 关键指标要用人工标注样本校准 judge，检查 judge 与人工判断的一致性。
- 高风险场景不能只依赖 LLM judge，需要规则校验或人工复核。

评分不是一个总分，而是**按维度拆开**。不同场景关心的维度不同，以任务创建场景为例（用户说"帮我建个明天早上去跑步"）：

| 评分维度 | 评什么 | 怎么判 | 满分条件 |
|---------|--------|--------|---------|
| 意图识别 | 是否正确识别为"创建任务" | 规则：检查是否调了 create_task | 调了 = 1 分，没调 = 0 分 |
| 参数提取 | 时间、标题是否正确 | 规则：日期 = 明天、时间段 = 早上、标题含"跑步" | 全对 = 1，部分对 = 0.5，全错 = 0 |
| 回复质量 | 回复是否自然、信息完整 | LLM 判分：按 rubric 打 1-5 分 | 4 分以上算通过 |
| 安全性 | 是否做了不该做的事 | 规则：是否调了非预期工具 | 无 = 通过，有 = 一票否决 |

优先从最容易出错、影响最大的维度开始。意图识别和参数提取比回复质量重要得多——AI 说话不够好听是小事，任务建错了是大事。

**3. 基线与对比（怎么用）**

Eval 的核心价值不是单次跑分，而是**对比**：改了 prompt / 换了模型 / 调了工具之后，分数是涨了还是跌了。没有基线的 eval 就是一次性实验，有基线的 eval 才是工程实践。

## 四、失败样本收集

失败样本是 eval 数据集最有价值的来源。核心思路：**在用户和 AI 交互的链路上埋点，捕获"AI 做错了"的信号**。

### 信号来源

| 信号 | 含义 | 怎么捕获 |
|------|------|---------|
| 用户纠正 | 用户说"不对，我的意思是……" | 对话中检测纠正意图（关键词或 LLM 分类） |
| 用户重试 | 同样的意思换个说法再问一遍 | 短时间内相似请求检测 |
| 工具调用失败 | agent 调了工具但参数错误或执行失败 | 工具层返回 error 状态 |
| 用户拒绝 | AI 建议创建任务，用户说"不要" | 确认流程中的拒绝动作 |
| 路由错误 | 问天气的请求被分给了 task agent | tracing 日志 + 人工抽检 |

### 收集结构

每条失败样本记录三部分：**输入**（用户说了什么）+ **错误输出**（AI 做了什么）+ **期望输出**（应该做什么）。前两个可以自动记录，第三个有时需要人工标注，有时可以从用户纠正中推导。

### 分析方式

优先级从高到低：

1. **规则先行**：用简单规则捞明显失败（工具返回 error、用户短时间重试、用户说"不对"）。成本低、延迟低，不需要 LLM。
2. **LLM 辅助分析**：对规则捞不到的灰色地带（AI 回复了但质量不确定的），用 LLM 异步批量分析。但 LLM 判 LLM 也不是 100% 准确，作为补充而非唯一手段。
3. **人工抽检**：高风险场景或 LLM 判分不确定时，人来最终判定。

### 冷启动策略

没有线上数据时，先手动构造——把自己使用过程中遇到的"AI 理解错了"的例子记下来。不需要一开始就有完美的自动化收集。大部分项目卡在第一步——数据都没记，后面什么都做不了。

## 五、Eval 在开发流程中的位置

### 与传统测试的对比

| | 传统测试 | Eval |
|---|---------|------|
| 触发时机 | 代码变更时 | 任何可能影响 AI 行为的变更（prompt、模型、工具 schema、路由逻辑、关键依赖） |
| CI 集成 | 测试失败阻止合并 | 按类型分层（见下方） |
| 数据集 | 测试用例（代码写死） | eval dataset（独立维护，持续积累） |
| 判定标准 | 通过 / 失败 | 分数 / 达标率 |

### 工程隔离

Eval 调用真实 LLM，速度慢且有成本，必须和普通测试隔离。推荐用目录隔离 + 不同的 npm script：

```
src/__tests__/
  unit/           # 普通单元测试
  integration/    # 普通集成测试
  eval/           # eval 测试（调真实 LLM）
```

```json
{
  "test": "vitest run --exclude='**/eval/**'",
  "test:eval": "vitest run src/__tests__/eval"
}
```

日常开发跑 `pnpm test`，不触发 eval。需要时跑 `pnpm test:eval`。CI 里把两者分成不同的 job。

**CI 阻塞策略分层：**

| Eval 类型 | CI 策略 |
|----------|---------|
| 核心意图识别、工具选择、参数提取 | 分数下降应阻塞合并 |
| 权限、安全、越权工具调用 | 失败应阻塞合并 |
| 开放式回复质量、语气、表达 | 可先告警或人工确认 |
| 成本、延迟、成功率 | 超过阈值时阻塞或告警 |

核心原则：会造成错误操作、越权、数据泄露或明显功能回归的 eval 应阻塞；主观质量类 eval 可以先报告。
