# 项目升级路线图与测试审计

> 当前状态：审计中；工程基线与 CI 质量门已接入  
> 创建日期：2026-05-23  
> 适用范围：`packages/web`、`packages/server`、`packages/shared`  
> 核心原则：先建立可回归的工程基线，再补齐现有产品闭环，然后生产化单 Agent，最后扩展客户端、语音、多 Agent 与 IoT。

## 1. 目标

本项目同时服务两个目标：

1. **真实可用的生产级个人项目**
   - 用户主要是项目作者本人，但功能必须经得起真实使用。
   - 重点是稳定、可恢复、少误操作、可回归、可观测。

2. **展示 LLM 工程能力的作品级项目**
   - 不只展示“接了一个 LLM API”，而是展示完整工程闭环：
     - eval 基准
     - prompt 管理
     - tool 边界
     - 上下文与会话管理
     - 高风险动作确认
     - token / 成本 / 延迟治理
     - tracing / 日志 / 失败样本复盘

这两个目标不冲突。优先级是：**先生产级，再把生产化过程沉淀成作品级表达**。

  - 建立业务场景覆盖矩阵

## 3. 测试审计方法

测试审计不是单纯追求覆盖率数字，而是回答：

1. 当前哪些业务行为已经有测试保护？
2. 哪些高风险行为没有测试保护？
3. 哪些测试只测了实现细节，没有覆盖用户可观察行为？
4. 哪些测试不稳定、不适合 CI？
5. 后续重构时，哪些模块最容易回归？

推荐顺序：

1. 盘点已有测试文件。
2. 按业务模块列出关键场景。
3. 将场景映射到 Route / Service / UI / Agent。
4. 标记已有测试、缺失测试、测试类型。
5. 按风险分为 P0 / P1 / P2。
6. 决定第一批补测范围。

## 4. 协作方式

### Codex 负责

- 静态阅读代码。
- 盘点测试文件。
- 运行项目内已有 `pnpm` 脚本。
- 建立测试覆盖矩阵。
- 标记缺失场景与风险等级。
- 提出补测顺序。
- 后续按确认后的计划补测试。

### 用户辅助

- 确认真实业务预期。
- 明确高风险动作的产品规则。
- 在需要真实邮箱、真实登录、浏览器交互、第三方服务时协助验证。
- 对 UI / 响应式 / 实际使用体验给反馈。

### 推荐模式

先自动审计，再手工检查。

不建议一开始就逐个页面手动点击，因为那样容易变成凭感觉验收，最后没有可追踪结果。更好的方式是：

1. 先用代码和测试建立风险地图。
2. 再针对高风险和 UI 强交互场景做手工验收。
3. 手工发现的问题再沉淀为测试或验收清单。

### 5.1.2 CI pipeline 审计结果

当前只有 `.github/workflows/deploy-cloudflare.yml`，但已经承担核心 CI / CD 职责：

- 触发条件：`pull_request` 到 `main`、`push` 到 `main`、`workflow_dispatch`。
- PR 阶段：只执行质量检查，不部署生产环境。
- `main` push / 手动触发：先执行质量检查，通过后部署 API 和 Web。
- CI job 当前执行：shared lint / typecheck、server lint / typecheck / test、web lint / typecheck / build。
- 部署保护：`deploy_api` 和 `deploy_web` 已限制为非 PR 事件，避免 PR 误触发生产部署。

判断：项目已经具备核心 CI 质量门。严格禁止不合格 PR 合并，还需要在 GitHub 仓库的 Branch protection 或 Ruleset 中把 `ci` job 设置为 required status check。

### 5.1.3 Web 测试基础设施审计结果

`packages/web` 当前只有 `dev`、`build`、`typecheck`、`lint`、`preview` 脚本，没有测试脚本。

当前缺失：

- React 组件 / 页面流程测试运行时。
- Web smoke test。
- 路由保护、登录态、表单校验、任务创建弹窗、AI 页面 SSE 行为的自动化测试。

判断：Web 不需要一开始追求大覆盖率，但应该先补最小 smoke test 和 2-3 条高价值流程测试，否则后续响应式和页面重构只能靠手工点击回归。

## 6. 测试类型约定

| 类型 | 用途 | 适合覆盖 |
| --- | --- | --- |
| 单元测试 | 验证纯规则和小模块 | 时间规则、冲突检测、输入校验、工具参数处理 |
| Service 测试 | 验证业务规则和数据库状态变化 | 任务 CRUD、权限、重复任务、群组关系 |
| Route 测试 | 验证 HTTP 契约 | 状态码、输入校验、鉴权、响应结构 |
| 前端组件 / 流程测试 | 验证用户可见行为 | 表单校验、按钮状态、页面跳转、错误提示 |
| Agent eval | 验证真实或模拟 LLM 行为 | 意图识别、工具调用、追问、拒绝、确认、高风险动作 |
| 手工验收 | 验证真实外部依赖和体验 | 邮箱、真实浏览器录音、响应式、移动端体验 |

## 7. 补测优先级规则

### P0：必须优先补

- 可能导致数据丢失。
- 可能导致越权访问。
- 可能导致 AI 误执行高风险动作。
- 当前计划中会被重构或增强的核心路径。
- 生产真实使用中高频出现的路径。

### P1：重要但可排后

- 影响主要体验但不会造成严重数据风险。
- 页面交互不完整。
- 错误提示不一致。
- 中频业务路径。

### P2：最后处理

- IoT / ESP32。
- 低频管理功能。
- 展示优化。
- 非主线部署能力。
