# AI 应用开发的底层心智模型（阶段性总结）

> 目标不是记住条目，而是建立一组"接到需求时自动展开的开场问题"——等价于后端开发里那种"听到需求脑子里自动有方向"的直觉。本文是那套直觉的显式编译版。

---

## 一、四层模型（L0 ~ L3）

| 层 | 名称 | 是什么 | 对应的工程问题 |
|---|---|---|---|
| **L0** | 原子层 | 一次 LLM 调用。三个面：输入（system/context/user 的拼装，即 prompt）、调用本身（选模型、参数、是否流式）、输出（纯文本 / 结构化输出 / 工具调用请求） | "这一次调用该怎么发、出什么" |
| **L1** | 编排层 | 多次 L0 怎么组合。**Workflow**（代码持图，把 L0 排成节点/分支）vs **Agent**（模型持循环，自己决定下一步、调几次、何时停） | "一次不够，怎么串 / 怎么循环" |
| **L2** | 上下文工程层 | 往窗口里放什么、从哪来。RAG（检索）、memory（短期会话 / 长期）、context 压缩与窗口管理 | "这次决策需要哪些信息、信息在哪" |
| **L3** | 横切层 | 包在外面观测和约束 L0 的东西。eval、可观测性 / tracing、成本与 token 核算、guardrails / 输出校验、缓存、错误处理与重试、流式 UX | "它是概率的且花钱，怎么知道它对、怎么兜住它错" |

**一个概念合并**：结构化输出和 tool calling 底层是同一能力（都是"逼模型按规定的格式输出"，即 function calling）。结构化输出＝强制模型按 schema 填；工具调用＝模型自己决定吐一个符合 schema 的调用。当成一回事，心智模型省一个格子。

---

## 二、层间关系：一个圆心 + 三个生长方向

这四层**不是 OSI 那种 L0 撑 L1、L1 撑 L2 的栈**。结构是：**L0 在圆心，L1/L2 是它两个无法自答的问题的答案，L3 是它一个固有属性的后果。**

- **圆心 = L0**：唯一的原子。它有三个面，每个面被"放大到需要独立处理"时，就长出一层。
- **L2 = L0 的输入问题长大后的样子**：当要塞进窗口的东西太大、会变、需要检索、需要跨调用保留时，"输入"就不再是一句手写 prompt，而要一个子系统（RAG / memory / 压缩）。
- **L1 = L0 的递归组合**：Workflow 把 L0 排成图，Agent 把 L0 排成循环。它没引入新原子，只引入"结构"。
- **L3 = 为"原子不确定 + 昂贵"付的税**：假如 LLM 调用是确定性的，L3 会大半塌回普通后端工程——会有单元测试而非 eval、普通日志而非"推理过程留痕"、固定成本而非 token 核算。L3 不是由 L0 组成的，它是**包在外面**观测和约束 L0 的。

---

## 三、关键耦合：L1 ↔ L2 通过 context 咬合

这是整张网里最重要的一条边。

**Agent 循环（L1）是制造 context 压力的源头**：每循环一步，历史累积一次，窗口无界增长，于是 memory / 压缩（L2）从"可选优化"变成"必须"。而固定 Workflow 里，每个节点看到什么由代码精确控制，context 压力小得多。

由这条边派生出三条后果，全部解释了"**能用 Workflow 就别上 Agent**"这条准则的根：

- Agent 比 Workflow **更难 eval**：要评估整条轨迹，而非单步。
- Agent 比 Workflow **更难观测**：要把每一步的调用链路（trace）记全，而不只是看单步。
- Agent 比 Workflow **成本更不可预测**：循环次数事先未知。

**一句话**：从 Workflow 走向 Agent，会同时抬高整个 L3 的税。所以跨过这条线要是"主动选择并接受代价"，不能被"agent"这个词无意识地推过去。

**补充（系统是分层判断的，不是非此即彼）**：一个系统可以"外层 Workflow + 某节点内层 Agent"。比如固定的"路由→回答→review→修改"是 Workflow 骨架，但"回答"节点内部若能自主调工具、自己决定调几次何时停，那个节点本身就是 Agent。控制流归属逐层判断。

---

## 四、接到 AI 需求时的四个开场问题（按优先级）

这是本文的核心。把它们当成接到需求时的默认展开动作，有意识地在每个项目上跑，直到像后端那套一样沉到潜意识。

1. **契约**：模型到底要产出 / 决定什么？我怎么判断它做对了？
   - 地位等同后端的"数据模型"，排第一。说不清"什么算对"，后面一切都没法可靠地搭。
2. **上下文**：为做这个决定，窗口里要放什么、从哪来？
   - 后端取数为了返回；AI 取数为了喂一个概率决策，且"该放什么"没有唯一正解，是判断。
3. **控制流归谁**：我的代码，还是模型？
   - 决定你在搭 Workflow 还是 Agent。注意"用了 LLM 做判断" ≠ "是 Agent"；区分线是**谁掌握控制流（循环、下一步去哪）**。一次 LLM 路由是在代码写死的图里选一条已有的边，选完控制权立刻回到代码，仍是 Workflow。
4. **它错了会怎样**：它一定会以某个概率错，这个预算开局就得留。

## 五、哪些问题能靠推演练，哪些必须靠构建

按问题的**性质**分，不是按编号顺序分。

### 判断 / 分类型 —— 大部分靠推演练，但有边界

- **Q1 契约**：答案在想清楚的那一刻就基本定了，不依赖运行结果。
- **Q3 控制流归谁**：**大部分情况**能推定。但在边界 case 上（"这个节点该 workflow 还是 agent"），只有在真实数据上跑完两个版本、看谁效果好才能定。典型案例：GitHub Copilot Code Review 2026 年从 workflow 切到 agent 架构，是试了两种才定的，不是推演定的。所以 Q3 有一条"在边界上必须靠构建验证"的尾巴。

> 练这两个收益是真的：推演在重复触发"识别问题形状 → 调取应对模式"这条回路，而直觉就是这条回路的自动化。但别高估推演的覆盖面——Q3 在清晰 case 上靠推演，在边界 case 上靠构建。

### 经验 / 概率型 —— 推演只能"把问题问全"，答不出实际取值，必须构建+测

- **Q2 上下文**：能推出"该放检索结果 + 最近几轮"这种结构；但"模型会不会真用、会不会被无关内容带偏、在哪类输入上崩"——只有跑了才知道。
- **Q4 错误形态**：能列出会出哪类错；但实际错误率、净收益正不正——只能 eval。

### 推演的正确用法与陷阱

- **把推演当"问题清单的编译器"，不是"答案的生成器"。** 练的是开场不卡壳、不漏维度。
- **危险**：推演完会产生"我想清楚了"的笃定，而这恰恰是 AI 里最不该有的笃定——概率行为无法预演。每次推演结束显式标注一句："Q1、Q3 我有答案；Q2、Q4 我只有假设，得构建才能验。"这句标注就是把"AI 没有确定性"也编译进直觉。
- **别只跑顺利路径**：专挑会把某一问逼到极端的需求（如明显该上 Agent 但成本会爆 → 逼 Q3 和 Q4 打架；"帮我写得更好" → 逼 Q1 契约难定义）。直觉的价值体现在权衡点，顺利路径上不需要直觉。

---

## 六、贯穿全篇的元认知

AI 应用里**永远不会有后端那种笃定的方向感**，这不是水平问题，是介质问题：后端的原子确定，能在脑子里推演确认方案可行；AI 的原子是概率的，推不出来，只能试了测。

所以"有方向"在 AI 里的定义本身变了——**不是"我有一个有把握的计划"，而是"我有一个假设 + 一个能快速验证它的办法"。** 接受这份"方向天然更临时"，反而更清晰，因为不再去追一个本就不存在的确定性。

**闭环路径**：推演刷回路、练全 Q1/Q3 的判断力 → 真实构建去填 Q2/Q4 的经验取值 → 复盘标注"这次哪一问最脆、为什么"。

# 第二部分：框架的第一个实战 —— "AI 互审"应用

> 这一部分不是把第一部分重讲，而是拿一个真实 idea 做完整回放：让一个最初想得过于乐观的需求，被第一部分的四个开场问题逐一拷问，看它修正后还剩什么。文中所有框架概念都回指上面的小节，不重复解释。

### 需求背景（记下来，免得忘了当初为什么要做）

**起因 / 痛点**：问稍微复杂的问题时，哪怕最强的单个模型也会出错。实践中反复观察到一个现象——让另一个 AI 把答案 review 一遍，质量明显变好。最直接的证据：写文档 / 代码时，Claude Code 写完、codex review，往往能挑出一些问题，而且这些建议大部分经我判断是真问题、值得采纳。于是想把这个"换个模型复核一遍"的动作固化成一个应用。

**用户故事**：作为使用者，我发出一个问题后，希望系统自动选一个最合适的模型来回答、再让别的模型复核并提出反对意见、然后让回答方据此修改，最终给我一个比"单次直接问"更可靠的答案；复核的深度（要几轮）我能在 UI 上自己定。

**当初设想的形态（即本次回放的起点，原貌保留，不用修正版盖掉）**：
- 一个"**管理 agent**"按问题挑最合适的模型来回答——当时认为这就构成"多 agent"。
- 用"**多 agent + 多轮冗余**"来提升可靠性，直觉是"多个模型同时出错的概率可以忽略"（`p₁×p₂×p₃` 很小）。
- 甚至让管理 agent 兼任"**最终复核者**"，以为这样错误率几乎可以压到零。
- 调用**多少轮、多深**，交给管理 agent 去判断和管理。

> 这一版的每一条，下面都会被四问拷问、并在 §九对照表里给出"改成了什么、为什么改"。

## 七、讨论后站得住脚的认知（地基）

每条是一个可以直接当设计前提用的结论，后面跟着"所以设计上意味着什么"。

1. **冗余只对"独立疏忽类"错误有效，对"共享盲区类"错误几乎无效。**
   `P(全错)=p₁×p₂×p₃` 只在错误**相互独立**时成立；而多个 LLM 面对同一问题，错误**高度相关**（共享语料、共享 prompt、共享架构）。最危险的是：越是"最强模型都会错"的难题，错误相关性越高——你最需要救援时，冗余恰好最失灵。
   → **设计含义**：别指望靠堆 reviewer 数量或加轮次把错误率"连乘式"压到趋零。价值来自"质"（独立视角），不来自"量"。

2. **盲区分两类，解药不同。**
   - *姿态性盲区*：知识其实在模型里，只是默认行为姿态让它不说（一味迎合提问者，即 sycophancy；以及看到现成答案就倾向认可的确认偏误）→ 靠改 reviewer 的**站位**能撬。
   - *知识性盲区*：正确答案根本不在任何模型里，或所有模型同样地错 → 提示工程层面**无解**，只能靠**模型之外的信息**（检索 / 执行验证）。
   → **设计含义**：reviewer 要两条腿——既会"换站位"，又能"调外部验证"。

3. **"独立视角" = 认知起点独立，不是嘴上要求"请独立思考"。**
   让 reviewer 一上来就看着成品答案审查，它会被锚定、顺着原答案找小毛病，放过"整条思路从一开始就歪了"这类最致命的错。让它**先不看初版答案、自己独立解一遍、再比对分歧**，相关性远低（切断了"共享 prompt"这条根）。
   → **设计含义**：reviewer 的工序是"先独立答 → 再比对分歧"，不是"接收答案 → 挑刺"。

4. **只有跨家族才削相关性，同厂自审基本没用。**
   工作方和 reviewer 要来自不同训练谱系（不同厂商），才真正削弱"共享语料/架构"那条根。
   → **设计含义**：模型选择上强制跨家族搭配。

5. **投毒是冗余的天然克星，无法根除、只能缓解。**
   训练时的语料污染会制造**跨模型共享**的知识盲区（换模型也救不了，正好是冗余的死穴）；为治知识盲区引入的外部检索，又新开了"推理时上下文注入"（检索回来的网页带毒）这个攻击面。
   → **设计含义**：外部检索按"默认带毒"来设计——宽检索 + 对结果持怀疑，目标是缓解不是根除。

6. **价值有明确边界。**
   在"错误以**独立疏忽**为主、且有**跨家族复核**"的任务上真有用（代码 / 文档——已亲身验证：Claude Code 写、codex review 能挑出真问题且大多被采纳）；在"错误以**共享知识盲区**为主"的任务上基本没用。
   → **设计含义**：定位成"代码/文档类的跨家族互审工具"，不是"通用 AI 校验器"。

### 哪些是"已定"、哪些是"待验"（这条分界来自 §五）

- **已定**（靠推演就定死，属判断型 Q1 契约 + Q3 控制流）：系统是什么形态、跨不跨家族、reviewer 先独立答、外部验证插哪——架构层面是确定的，可以照着建。
- **待验**（推不出，属经验型 Q2 上下文 + Q4 错误）：修改方到底听不听 review、多轮会不会越改越差、这一整套相对"单次问答"的**净收益正不正**——这些只能构建后用 eval 回答。**"这应用到底值不值得做"本身就落在待验区。**

## 八、顺着地基重新长出来的需求（施工图）

粒度停在数据流 + 角色职责 + 控制流 + 外部验证位置 + 如何判断有用；不下探到每个角色的输入/输出契约（那是真开工时才落的活，现在写=未验证的猜测，违背 §五"经验型问题别假装推得出"）。

**整体形态**：外层是一条代码持图的固定流水线（Workflow），其中"回答"节点内部是 Agent。（即 §三补充的"外层 Workflow + 节点内层 Agent"。）

**数据流（四步直线）**：

1. **路由** —— 一次轻量 LLM 分类（或几条规则），按问题类型选一个工作 agent。这是 Workflow 的 routing 节点：模型在节点内做了选择，但选完控制权立刻回到代码，**不是 Agent**（§四 Q3）。
2. **回答** —— 选中的工作 agent（某家族）解题。若它能自主调工具（搜索 / 写文件，自己决定调几次、何时停），**这个节点就是内层 Agent**（§三补充）。
3. **独立复核** —— 一个**不同家族**的 reviewer，先不看初版答案、独立解同一题，再比对两份答案、输出分歧点（地基 3、4）。
4. **修改** —— 工作 agent 把分歧点当"不同意见"做**有约束**的修改（prompt 里规定怎么权衡，避免盲从 review、也避免阳奉阴违）。

**控制流归谁**：整条线的步数 / 轮次由"用户在 UI 定的深度参数 + 你的代码"控制，**不交给管理 agent**。理由是 §四 Q3 + §三：用户已定死深度，循环次数就是个确定的数，交给 agent 等于平白付 §二的 L3 税（不可预测、难 eval），且换不来确定性。每个节点只负责把单步做好。

**外部验证插在哪**：插在 reviewer（以及可选的工作 agent）能调检索 / 执行去查证事实的能力上。这是**唯一能碰知识性盲区的杠杆**，价值排在所有话术设计之前（地基 2、5）。检索默认带毒，宽检索 + 存疑。

**怎么判断它有用（待验区，对应 Q2/Q4）**：开局就备一组"你被最强模型坑过"的真实难题（代码 / 文档为主）当 eval 集，对比"单次问答 vs 走完整套"。**最关键记一栏：review 抓到的、且是你自己原本也会漏的问题——只有这一栏是系统的真实净产出**（你本来就会发现的不算它的功劳）。没有这个记录，整套系统就是个没法验证、只能凭感觉相信它有用的东西（§一 Q1 + §五）。

## 九、对照：最初设想 → 修正版 → 为什么改

| 最初设想 | 修正版 | 为什么改（依据） |
|---|---|---|
| 多 agent + 多轮冗余，把错误率连乘式压到趋零 | 跨家族互审，抓"独立疏忽类"错误；错误率**不**连乘 | 错误高度相关，连乘公式前提失效（地基 1） |
| "管理 agent 选哪个 LLM"算多 agent | 它是 Workflow 的 routing 节点 | 用了 LLM 判断 ≠ Agent；控制流没交给模型（§四 Q3） |
| 管理 agent 当最终复核者、不够再加一层 | 砍掉这个想法 | 会陷入"谁来审复核者"的无限套娃（turtles all the way down）：堆 reviewer 换不来确定性（地基 1、§三） |
| reviewer 接收成品答案后审查 | reviewer 先独立解题、再比对分歧 | 认知起点独立才算"独立视角"（地基 3） |
| 轮次交给 agent 自己判断、我兜底 | 轮次交给代码 + 用户参数 | 别为不必要的自主性付 L3 税（§二、§四 Q3） |
| 多个 LLM 同时出错概率可忽略 | 难题恰是共享盲区最密、冗余最失灵处 | 错误相关性，尤其在难题上趋近 1（地基 1） |
| 感觉"哪类问题都能用" | 边界：代码 / 文档这类"独立疏忽为主"的任务才有用 | 盲区结构决定能力边界（地基 6） |

## 附录 A：认知校准 — 传统开发者的 5 个直觉错位

> 从传统 Web 开发进入 AI 应用开发，最迷糊的不是"不知道概念"，而是直觉对不上。做项目时撞见困惑，回来看这 5 条能快速定位"我的哪个直觉失灵了"。

### 错位 1：从确定性到概率性

Web 后端相同输入 → 相同输出，可 assertEquals。LLM 相同输入 → 可能不同输出，只能用 Eval 测分布。

### 错位 2：LLM 是无状态纯函数

LLM 不记得任何事。所有"记忆"是调用方每次把历史塞进去。类比 HTTP 无状态 + cookie。所谓 "agent memory"、"对话上下文"，本质都是调用方在管理 state。

### 错位 3：Context Window 是新资源

CPU / 内存 / 带宽之外多了一种：context window（按 token 计）。它同时是容量约束、成本变量、质量变量（塞太多反而变笨）。

### 错位 4：控制流倒置

传统：你调库。Tool calling：你把函数描述给 LLM，LLM 决定调哪个、传什么参数——IoC。你的业务代码被概率组件调度。

### 错位 5：失败模式隐蔽

传统失败：异常 / 500 / 超时，显式。LLM 失败：输出看起来对但其实错（幻觉、推理错误、漂移）。没有 Eval + Observability 你发现不了。

## 附录 B：实操默认值速查

> 经验性的起手默认值，不是教条。熟练后你会有自己的判断。

**模型与调用**

- 默认最小够用模型起步，不够再升
- Temperature 默认 0；需要创造性才调高
- 下游要解析 → 永远用 SDK 原生 structured output

**Prompt**

- 四要素：角色 + 任务 + 上下文 + 输出格式
- 任务要具体（"做 X、不做 Y"）

**上下文**

- 能不进 context 就不进
- 利用开头/结尾位置效应（lost in the middle）
- 超窗口：裁剪 → 摘要 → 外置记忆（优先级递增）

**Workflow 节点**

- 确定性逻辑不交给 LLM
- 一个节点只做一件事
- 节点间用结构化 schema，不用自然语言字符串

**Agent / 工具**

- 工具 5-10 个，少而正交；命名和描述就是 prompt
- 终止条件三层并联：完成信号 + 步数上限 + 成本上限
- Context 必爆：截断旧结果 / 摘要 / 外存

## 附录 C：术语速查

> 遇到再查，不用一次记住。

| 术语 | 一句话 |
|------|--------|
| Token | LLM 处理的最小单位，≈ 0.75 英文词 / 1.5 汉字 |
| Context window | 单次调用能处理的最大 token 数 |
| Structured output | 让 LLM 按指定 schema 输出 |
| Temperature | 控制输出随机性（0 最稳，1+ 创造性强） |
| System prompt | 定义 LLM 角色和行为的开场指令 |
| Few-shot | 在 prompt 里给几个示例教 LLM |
| RAG | 检索增强生成，从外部知识库取相关内容塞进 context |
| Embedding | 把文本转成向量，用于语义搜索 |
| Vector DB | 存储 embedding 的数据库 |
| Memory | 跨调用保留的信息（短期对话 / 长期档案） |
| Prompt chaining | 串行多个 LLM 调用，前一步输出喂下一步 |
| Routing | 先分类再分发到不同处理器 |
| Parallelization | 并行调用合并结果 |
| Orchestrator-workers | 一个 LLM 规划 + 多个 LLM 执行 |
| Evaluator-optimizer | 生成 + 评估 + 改进的循环 |
| Tool / Function calling | 让 LLM 调用预定义函数 |
| Agent loop | think-act-observe 的循环 |
| ReAct | Reasoning + Acting 的经典 agent 模式 |
| Termination condition | 判断 agent 任务完成的条件 |
| Eval | 评估系统，测试 LLM 输出质量 |
| Trace | 一次完整调用的全链路记录 |
| Guardrails | 输入/输出的护栏 |
| MCP | Model Context Protocol，工具/数据源标准化协议 |
| A2A | Agent-to-Agent 通信协议 |
| Hallucination | LLM 自信地编造不存在的事实 |
| Prompt injection | 通过用户输入操纵 LLM 行为的攻击 |
