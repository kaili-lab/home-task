# AI 应用开发

> 本目录用于梳理 AI 应用从搭建到产品化的完整能力地图。当前只保留目录级 README，后续确认结构后再展开具体专题。

## 能力地图

AI 应用开发拆成两大部分：**应用搭建能力**（如何做出来）和**产品化能力**（如何稳定运行）。

```
AI 应用开发
├── 应用搭建能力
│   ├── 任务定义与成功标准
│   ├── Eval 驱动开发
│   ├── 模型选型策略
│   ├── Prompt / Context Engineering
│   ├── RAG / 记忆 / 数据接入
│   ├── Tool / Function Calling 设计
│   ├── Agent loop / workflow
│   ├── 多 Agent 编排
│   └── 产品交互设计（含多模态）
└── 产品化能力
    ├── Eval 与数据飞轮（线上回流与回归基线）
    ├── Observability 与 Tracing
    ├── 安全、权限与隐私
    └── 成本、延迟与可靠性（含错误处理与降级）
```

## 应用搭建能力

关注"如何把 AI 应用做出来"，包括模型如何选择和接入、上下文如何组织、工具如何设计、流程如何编排、用户如何交互，以及在搭建期就要建立的 eval 反馈循环。

它不等同于 Agent 构建。Agent 是一种组织方式，多 Agent 是更复杂的组织方式。很多 AI 应用的核心难点其实在 context、RAG、工具边界、状态管理和产品交互。后续展开时，应避免默认把复杂度推向多 Agent——优先判断问题是否能通过更清晰的上下文、更好的工具设计、更确定的 workflow 或更好的交互完成。

**关于 Eval 的横跨定位：** Eval 不只是上线后的事。搭建期需要先有一个最小可用的 eval 集，prompt、RAG、tool 设计的每一次迭代都要靠它给信号——这是「Eval 驱动开发」。线上之后再通过用户反馈和失败 case 持续扩充 eval 集、形成「数据飞轮」，这部分归在产品化能力。两阶段共享同一套基础设施，但服务于不同目标。

详见：[应用搭建能力](./应用搭建能力/README.md)

## 产品化能力

关注"如何让 AI 应用稳定、可评估、可观测、可治理地运行"，包括 eval、数据飞轮、tracing、安全权限、隐私、成本、延迟和可靠性。

这些能力决定 AI 应用能否从 demo 走向真实产品。

详见：[产品化能力](./产品化能力/README.md)
