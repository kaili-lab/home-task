# 应用搭建能力

> 应用搭建能力关注“如何把 AI 应用做出来”。它是 AI 应用的基础层，但不应该被简化为 Agent 构建和多 Agent 编排。

## 能力范围

当前建议先按以下主题理解：

| 主题 | 关注点 |
|------|--------|
| 场景建模 | 明确用户目标、任务边界、成功标准和不可做事项 |
| 模型选型策略 | 选什么模型（能力 vs 成本 vs 延迟）、什么时候用大模型什么时候用小模型、是否 fine-tune、模型抽象层设计 |
| Prompt / Context Engineering | 组织系统指令、用户输入、历史上下文、业务规则和外部信息 |
| RAG / 记忆 / 数据接入 | 数据切分、召回、重排、权限过滤、引用溯源和过期数据处理 |
| Tool / Function Calling 设计 | 工具粒度、参数 schema、幂等性、确认流程和错误恢复 |
| Agent loop / workflow | 让模型在可控流程中观察、决策、调用工具和停止 |
| 多 Agent 编排 | 在确有必要时拆分角色、上下文、工具权限和 handoff 契约 |
| 产品交互设计 | 决定什么时候自动执行、什么时候追问、什么时候确认、什么时候拒绝；包括多模态交互（语音、图像等） |

## 对 Agent 的定位

Agent 是一种组织 AI 能力的方式，不是 AI 应用的全部。

单 Agent 适合职责清晰、工具数量可控、上下文规模适中的场景。多 Agent 适合不同角色确实需要不同上下文、工具权限或并行协作的场景。

如果问题可以通过明确 workflow、缩小工具边界、改善上下文或优化交互解决，就不应优先引入多 Agent。

## 主要难点

应用搭建阶段的难点通常包括：

- 如何选择合适的模型，在能力、成本和延迟之间做取舍。
- 如何把业务问题拆成模型适合处理的任务。
- 如何给模型足够上下文，但不让上下文污染或膨胀。
- 如何设计工具，让模型容易正确调用，系统也能安全执行。
- 如何处理多轮状态、用户偏好、长期记忆和任务进度。
- 如何让 AI 在不确定时追问，而不是猜测。
- 如何判断哪些逻辑交给模型，哪些必须由确定性代码控制。
