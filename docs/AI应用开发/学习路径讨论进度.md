## 用户目标（已确认）

1. **同时想要两样东西**：
   - **A. 思考套路**：拿到 AI 应用需求时，脑海里有清晰的拆解顺序（类比传统全栈的"需求→建模→模块→CRUD"）。

3. **学习方式偏好**：学习阶段刻意慢下来，亲自参与构建甚至手敲每一行代码（区别于工作代码用 AI 加速）。

## 已对齐的方法论原则

### 「慢下来」的边界（已确认）

- **慢的对象**：AI 应用的**关键决策点和对比验证**——prompt 的每个字、tool schema 的每个字段、eval 的每条评分规则、不同参数/策略的对比实验。
- **不慢的对象**：脚手架代码（HTTP server / UI / ORM / 登录等），AI 生成模板能跑就行。
- **原则**：手敲的对象应该是"AI 应用核心决策"，不是"代码量"。

### 5 层框架的定位（轮次 2 重要修正）

- 5 层框架（L0-L4）= **技术地图**，不是**学习路径**。
- 它回答"AI 应用世界都有什么"，不回答"我下一步该做什么"。
- 把它当学习路径用，会产生副作用：
  - 暗示"L4 multi-agent"是终态目标（实际上 99% 真实应用停在 L2/L3）
  - 弱化横切关注点（Eval、可观测）的优先级
  - 不解决用户真正缺的「思考套路」（用户期待的是"按问题解决阶段"切的思路，不是"按技术组件"切的）
- **结论**：学习路径要重新校准，不直接套 5 层。但 5 层作为**地图**保留，用来在学习时"知道自己在哪"。

### 学习路径切分维度（轮次 3 确定）

**选定切分维度 3：按真实项目难度递增**。

3 个项目，难度递增：

- ~~难度 1：单次 query 解决~~（轮次 7 跳过，理由见下）
- 难度 1：多步组合（workflow）
- 难度 2：开放式探索（agent）
- 难度 3（可选）：多 agent 对比实验

**关键警戒线**：项目动机必须是"做这个东西"，不能是"练 agent loop"。否则会滑回"按技术层次走"，毕业后的判断力会偏。

### 每个项目的思考指引（轮次 3 提出，轮次 6 简化）

不做执行清单，只留两个方向性提醒：

- **开始前问自己**：这个项目做完怎么算好？怎么知道它错了？
- **结束后问自己**：我的哪个传统直觉在这个项目里对不上了？

具体拆多细、写不写下来，由用户在做的过程中自行决定。

### 项目选择方法论（轮次 5 确定）

不由 Claude 单方推荐具体项目，改为：

1. **Claude 给"项目特征清单"**（每种集成形式的理想项目应满足的特征）
2. **用户按特征 + 自己熟悉领域**提候选项目
3. **Claude 审核**候选是否符合特征、是否有坑、是否有更优变体

原因：用户的领域熟悉度直接影响 eval 标注速度和判断准确度——把"领域熟悉度"作为常数固定，让"AI 能力"成为唯一变量。

## 项目特征清单（选项目时对照用）

### ~~项目 1 — Single LLM Call~~（已跳过）

特征：

1. 在 ChatGPT / Claude 网页上发一条消息、收到一条回复，就是一次 single LLM call。
2. 多轮对话也是——每次发送时把历史拼进 context，仍然是一次调用
3. LLM 返回一次 tool call，执行后喂回结果，LLM 再给最终答案。这在 API 层面是 2 次调用，但逻辑上仍是"一次交互"，没有循环、没有分支

**跳过理由（轮次 7 确定）**：

- single LLM call 在日常 AI 工具（ChatGPT / Claude / AI CLI）中已无处不在，单独做项目学习价值低
- 其核心概念（structured output、eval 基础、task definition）可在 workflow 项目的单个节点中练到
- **注意**：做 workflow 项目时，第一步先给其中一个节点建 eval（如 routing 分类节点），在单节点上理解 eval 闭环后再扩展到 workflow 整体

### 项目 2 — Workflow 应有特征

1. 任务自然分解为 2-5 步
2. **至少触发以下 patterns 中的 2 个**：routing / parallelization / evaluator-optimizer（不要只用 prompt chaining，太朴素）
3. 30 个真实输入样本可获取
4. 领域熟悉：能判断**每一步**输出质量，不只是最终输出
5. 有可触发"reliability cliff"的场景（人为让某节点失败 10%，体会 chain 崩塌）

### 项目 3 — Single Agent 应有特征

在 ChatGPT / Claude 网页上发一条消息、收到一条回复，就是一次 single LLM call。
多轮对话也是——每次发送时把历史拼进 context，仍然是一次调用。

1. 工具集少（3-5 个），每个工具边界清晰
2. **任务路径不可预定**（如果路径固定，就该用 workflow，不是 agent）
3. **结果客观可验证**（答案对/错明确，不能是主观评价）
4. 领域熟悉：能判断 agent **每一步的决策**合不合理（不只是最终答案）
5. 能撞见 agent 三大难点：终止条件、上下文管理、self-correction

### 项目 4 — Multi-Agent / 对比实验 应有特征

1. 任务有"宽度优先"特征（多角度独立调研，而非线性推理）
2. 单 agent 能做但效率/质量有可见瓶颈
3. 至少 5-10 个评测样本能客观对比版本间的成本/延迟/质量
4. 用户的多模型 peer review idea 可放进来作为对比版本之一

### Claude 的修订建议（供参考，可拒绝）

- **项目 1**：GitHub Issue → 结构化分类/抽取（数据公开 + 领域熟悉）
- **项目 2**：代码 PR 自动评审 workflow（routing + parallelization，Anthropic 官方典型例子）
- **项目 3**：代码仓库探索 agent / Codebase Q&A（工具极简 + 领域熟悉 + 答案客观）
- **项目 4**：多视角研究助手对比实验（A 单 agent / B 传统 multi-agent / C 多模型 peer review）

### AI 造数据的边界（轮次 5 确定）

**重要常识盲点**：

- ✅ 可以：作为 prompt few-shot 示例
- ✅ 可以：冷启动开发期的快速 mock
- ❌ 不可以：作为 eval 集（AI 造的样本缺少真实噪声，eval 通过 ≠ 真实场景能用）
- ✅ 折中：AI 造 30 条 + 手工真实样本 10-15 条作"硬样本"，最终质量评估只看真实样本子集

### 项目 4 形式（轮次 5 确定）

不做"一个 multi-agent 产品"，做"对比实验"：

- **A 版（baseline）**：单 agent + 多次工具调用
- **B 版（传统 multi-agent）**：lead + 并行 sub-agents（参考 Anthropic Deep Research）
- **C 版（用户的 idea）**：单 agent 主调研 + 多模型 peer review + 反思（本质上仍属 Evaluator-Optimizer workflow，但用了多模型）

三版对比成本/延迟/质量，让用户得到"何时该用 multi-agent / 多模型"的一手判断。

### 轮次 2（2026-05-25）

- 用户确认"慢下来"边界。
- 用户质疑 Claude 是否被 5 层框架限制。
- 用户提出候选学习项目：「看书讨论助手」（PDF/EPUB 向量存储 + 多 AI 协商讨论），增量路径"先回答→结构化→工具→知识库→eval→agent loop→确认→多 agent"。
- **Claude 反思承认**：是的，被 5 层限制了，且没意识到这是限制。线性 vs 螺旋两种形状本质都是"按层走"。
- **重要修正**：5 层是"技术地图"，不是"学习路径"。它不解决用户真正的痛点（按问题解决阶段切的思考套路）。

  - 优点：真实需求 + 增量路径感清晰
