## 已对齐的方法论原则

### 「慢下来」的边界（已确认）

- **慢的对象**：AI 应用的**关键决策点和对比验证**——prompt 的每个字、tool schema 的每个字段、eval 的每条评分规则、不同参数/策略的对比实验。
- **不慢的对象**：脚手架代码（HTTP server / UI / ORM / 登录等），AI 生成模板能跑就行。
- **原则**：手敲的对象应该是"AI 应用核心决策"，不是"代码量"。

### 5 层框架的定位（轮次 2 重要修正）

- 5 层框架（L0-L4）= **技术地图**，不是**学习路径**。
- 它回答"AI 应用世界都有什么"，不回答"我下一步该做什么"。
- 把它当学习路径用，会产生副作用：
  - 暗示"L4 multi-agent"是终态目标（实际上 99% 真实应用停在 L2/L3）
  - 弱化横切关注点（Eval、可观测）的优先级
  - 不解决用户真正缺的「思考套路」（用户期待的是"按问题解决阶段"切的思路，不是"按技术组件"切的）
- **结论**：学习路径要重新校准，不直接套 5 层。但 5 层作为**地图**保留，用来在学习时"知道自己在哪"。

### 学习路径切分维度（轮次 3 确定）

**选定切分维度 3：按真实项目难度递增**。

3-4 个项目，难度递增：
- 难度 1：单次 query 解决
- 难度 2：多步组合（workflow）
- 难度 3：开放式探索（agent）
- 难度 4（可选）：复杂场景

**关键警戒线**：项目动机必须是"做这个东西"，不能是"练 agent loop"。否则会滑回"按技术层次走"，毕业后的判断力会偏。

### 每个项目的强制仪式（轮次 3 确定）

把切分 1（问题解决阶段）和切分 2（认知错位）"内嵌"进项目驱动里——

**开始仪式（按切分 1 的 6 个阶段，写在项目 README）**：
1. 任务定义 + 可观察的成功标准
2. 输入输出 schema
3. 上下文/数据来源
4. "代码做的 vs 模型做的"边界
5. 至少 5 种可能的失败模式
6. 评估方案 + 至少 10 个人工标注样本

**结束仪式（按切分 2 的 5 个认知错位复盘）**：
1. "概率性"在哪让我吃过亏？
2. "状态由调用方持有"哪里没想清楚？
3. context 限制让我做了哪些妥协？
4. tool calling 哪里跟直觉不一样？
5. 最难发现的失败模式？

**不写不算结束**。

### 项目选择方法论（轮次 5 确定）

不由 Claude 单方推荐具体项目，改为：

1. **Claude 给"项目特征清单"**（每种集成形式的理想项目应满足的特征）
2. **用户按特征 + 自己熟悉领域**提候选项目
3. **Claude 审核**候选是否符合特征、是否有坑、是否有更优变体

原因：用户的领域熟悉度直接影响 eval 标注速度和判断准确度——把"领域熟悉度"作为常数固定，让"AI 能力"成为唯一变量。

### AI 造数据的边界（轮次 5 确定）

**重要常识盲点**：

- ✅ 可以：作为 prompt few-shot 示例
- ✅ 可以：冷启动开发期的快速 mock
- ❌ 不可以：作为 eval 集（AI 造的样本缺少真实噪声，eval 通过 ≠ 真实场景能用）
- ✅ 折中：AI 造 30 条 + 手工真实样本 10-15 条作"硬样本"，最终质量评估只看真实样本子集

### 项目 4 形式（轮次 5 确定）

不做"一个 multi-agent 产品"，做"对比实验"：

- **A 版（baseline）**：单 agent + 多次工具调用
- **B 版（传统 multi-agent）**：lead + 并行 sub-agents（参考 Anthropic Deep Research）
- **C 版（用户的 idea）**：单 agent 主调研 + 多模型 peer review + 反思（本质上仍属 Evaluator-Optimizer workflow，但用了多模型）

三版对比成本/延迟/质量，让用户得到"何时该用 multi-agent / 多模型"的一手判断。

- **能力主题目录**（任务定义 / Eval / 模型选型 / Prompt / RAG / Tool / Agent / 多 Agent / 交互）= **知识纵深**。

## 讨论历史

### 轮次 1（2026-05-25）

- 用户提出困惑：有传统全栈的清晰思路，但没有 AI 应用的拆解思路。
- 用户初步设想：按 5 层地图，每层手搭 demo。
- Claude 校准：5 层地图直接对应 B（学习路径），但不直接给 A（思考套路）。
- 用户确认：两个都要，且不限于当前项目，且偏好亲手构建。
- Claude push back："慢下来"的边界要校准——慢在关键决策点的对比验证，不是脚手架代码量。

### 轮次 2（2026-05-25）

- 用户确认"慢下来"边界。
- 用户质疑 Claude 是否被 5 层框架限制。
- 用户提出候选学习项目：「看书讨论助手」（PDF/EPUB 向量存储 + 多 AI 协商讨论），增量路径"先回答→结构化→工具→知识库→eval→agent loop→确认→多 agent"。
- **Claude 反思承认**：是的，被 5 层限制了，且没意识到这是限制。线性 vs 螺旋两种形状本质都是"按层走"。
- **重要修正**：5 层是"技术地图"，不是"学习路径"。它不解决用户真正的痛点（按问题解决阶段切的思考套路）。
- Claude 对"看书讨论助手"的初步评估：
  - 优点：真实需求 + 增量路径感清晰
  - 坑 1：把"多 AI 协商"作为终态是 5 层框架副作用的体现（多视角分析用单 agent 就能做，便宜 10-20 倍）
  - 坑 2："讨论得好"的成功标准模糊（任务定义这一步缺失）

### 轮次 3（2026-05-25）

- 用户选定切分维度 3（按真实项目难度递增）。
- Claude push back：用户的论证里藏着"按层走"的滑坡——动机必须是"做这个东西"而非"练 agent loop"。
- 确定每个项目的开始/结束仪式（内嵌切分 1 和切分 2）。

### 轮次 4（2026-05-25）

- 用户要求 Claude 调研典型项目案例后推荐 3-4 个项目。
- Claude 派 general-purpose agent 做调研，关键发现：
  - 真正"必须用 multi-agent"的工业案例几乎只有 Deep Research 一类
  - single agent / workflow 的边界正在模糊（GitHub Copilot Code Review 2026 年切换案例）
  - reliability cliff（95% × 3 = 86%）是 workflow 学习真正动机
- Claude 推荐 4 个项目：会议→行动项 / PR 评审 workflow / 数据分析 agent / multi-agent 对比实验
