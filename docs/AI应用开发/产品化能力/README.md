# 产品化能力

> 产品化能力关注“如何让 AI 应用稳定、可评估、可观测、可治理地运行”。它决定 AI 应用能否从 demo 走向真实产品。

## 能力范围

当前建议先按以下主题理解：

| 主题 | 关注点 |
|------|--------|
| Eval 与数据飞轮 | 用数据集、评分、基线和线上反馈持续衡量并改进 AI 行为 |
| Observability 与 Tracing | 记录模型调用、工具调用、RAG、路由和最终回复，用于失败定位和质量分析 |
| 安全、权限与隐私 | 控制工具权限、数据访问、prompt injection、PII、审计和高风险操作 |
| 成本、延迟与可靠性 | 控制 token、模型路由、缓存、超时、成功率和 SLO；包括错误处理与降级策略（模型超时兜底、调用失败重试、graceful degradation） |

## 与测试体系的关系

测试体系属于产品化能力的一部分，但产品化能力不等于测试体系。

更准确的边界是：

- Eval 与数据飞轮：测试体系的核心组成。
- Observability 与 Tracing：测试体系的重要支撑，负责失败归因和线上质量分析。
- 安全、权限与隐私：主体是治理能力，测试体系覆盖其中可验证、可回归的部分。
- 成本、延迟与可靠性：主体是工程与运营能力，测试体系覆盖性能回归、预算阈值和降级路径。

## 主要难点

产品化阶段的难点通常包括：

- 如何定义 AI 行为质量，而不是只凭主观体验。
- 如何把线上失败转化为 eval case 和改进输入。
- 如何定位失败发生在意图、路由、检索、工具、权限还是生成。
- 如何保证工具调用不会越权、误操作或泄露数据。
- 如何在质量、成本、延迟之间做可验证的取舍。
- 如何设计降级策略——模型超时、返回异常、调用失败时如何兜底，而不是直接报错。
- 如何让 prompt、模型、工具和数据变更都能被回归验证。
