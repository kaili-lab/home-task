# 文档工程方法对比指南

> 面向完全没接触过这些方法的开发者，从"为什么要有这个东西"开始讲。

---

## 先说一个根本问题：为什么要费心思"设计文档"？

开发者普遍认为写文档是浪费时间——代码才是真相，文档总会过期。这个感受是对的，但原因诊断错了：**过期的不是文档这件事，而是"用错了方式写文档"**。

下面四种方法，每一种都是在回答"怎么让文档不成为负担"这个问题，只是从不同角度切入。

---

## 快速对比

| 方法 | 解决什么问题 | 最小产出 | 国内 | 国际 |
|------|-------------|---------|------|------|
| Sequence Diagram | 多人/多模块的调用顺序难以靠读代码感知 | 一张时序图 | 很普及 | 极普及 |
| C4 Model | 不同受众需要不同粒度的架构视图 | L1+L2 两张图（复杂模块再补 L3） | 中等，增长中 | 高 |
| Diátaxis | 文档越写越多但越来越难用 | 重新分类现有文档 | 低 | 中高 |
| Living Spec | 规格说明写了就过期，开发者不信任它 | 一份随 PR 更新的 spec 文件 | 低（概念有，术语少） | 高（AI 时代爆发式增长） |

---

## 一、Sequence Diagram（时序图）

### 是什么

一种图，横轴是参与者（模块/服务/用户），纵轴是时间，箭头表示"谁在什么时候调用了谁、得到什么返回"。

### 为什么它存在（这是关键）

软件系统大了之后，一个功能的完成往往需要多个模块按特定顺序协作。这时候出现了一个经典困境：

- 看代码：你必须在脑子里"模拟执行"，一边追函数调用，一边记住"现在在哪个模块里"。模块多了，脑子就满了。
- 看文字描述：说"A调用B，B调用C，C返回结果给B，B再返回给A"——这是在用线性文字描述一个本质上是并发+顺序的过程，信息密度很低，还容易遗漏分支。

**时序图的本质洞察是：时间是一个维度，不能压平成文字。**

你能在半分钟内从时序图里获取的信息，可能需要读15分钟代码才能还原。而且时序图天然呈现"分支"（alt/else），这正是代码里最复杂的部分。

对于 AI agent 这种"LLM → tool call → 结果 → LLM → ..."的往复循环，时序图几乎是唯一合适的表达方式。

### 什么时候用

- 有 3+ 个模块/服务参与的流程
- 有条件分支（成功/失败/冲突）的流程
- 需要向新人解释"一个请求从进来到出去经历了什么"

### 什么时候不适合用

- 单模块内部的逻辑（用流程图/伪代码更好）
- 纯数据结构设计（用 ER 图）
- 覆盖所有边界情况（图会爆炸，那是单测的工作）

### 学习建议

1. **直接看 [Mermaid 官方文档](https://mermaid.js.org/syntax/sequenceDiagram.html)** — 15 分钟，语法量很小
2. **让 AI 生成第一张** — 描述你的流程，让 AI 输出 Mermaid 代码，自己再调整
3. **不要追求一次画完整** — 先画 happy path，再加异常分支

### 在 AI 编程中的价值与用法

**为什么 AI 特别善于处理时序图**

Mermaid 格式的时序图是结构化文本——不是模糊的散文，而是有明确语法的信息。AI 工具在 context 中读到它时，提取信息的准确率远高于等量的文字描述。

用文字告诉 AI"A 调用 B，B 查数据库，有冲突返回错误，没冲突写入"——AI 需要从散文里推断结构。同样的信息用时序图的 `alt/else` 表达，AI 直接得到结构化的分支树。通常信息密度更高，歧义更少。

**具体用法**

1. **Diagram-first 实现**：先画时序图描述你想要的流程，把 Mermaid 代码粘给 AI，说"按这张图实现"。比用自然语言描述的实现准确率高得多，因为 AI 不需要猜测调用顺序和分支逻辑。

2. **Bug 定位上下文**：遇到 bug 时，把"预期时序图"和"实际发生的行为"一起给 AI，AI 能更快定位"哪个箭头方向错了"或"哪个分支没有处理"。比单纯贴报错信息有效得多。

3. **放进 CLAUDE.md 或 .cursorrules**：把核心流程的时序图放在 AI 长期能读到的地方。这样 AI 在修改任何相关代码前，都有流程全图作为参照，不会无意中破坏调用链。

4. **让 AI 反向生成图**：对存量代码，让 AI 读代码并输出 Mermaid 时序图。这既是文档补全，也是验证——如果 AI 生成的图和你预期的不一样，说明代码和设计之间已经有了漂移。

5. **Review AI 生成的代码**：让 AI 生成代码后，让它同时生成对应的时序图。对照图 review 代码，比直接读代码更快发现"调用顺序错误"或"漏掉某个分支"。

**本项目的实际应用**

`Single-Agent设计文档.md` Section 4.2 是本项目 AI chat 模块的完整时序图。在修改 `agent-loop.ts` 或 `tool-executor.ts` 时，把这张图粘给 AI，它在理解完整流程的前提下修改代码，而不是自己猜 `skipSemanticConflictCheck` 这类状态的传递逻辑。

## 二、C4 Model（C4 架构图）

### 是什么

一套用四个层级描述软件架构的方法，从最粗到最细：**系统上下文（L1）→ 容器（L2）→ 组件（L3）→ 代码（L4）**。支持 Mermaid 语法，图可以和代码住在同一个仓库里。

### 为什么它存在（这是关键）

架构图有一个长期未被解决的问题：**"架构图"对不同人意味着完全不同的东西。**

- CTO 问"架构是什么"，想看的是系统和外部的边界
- 新入职开发者问同样的问题，想知道哪个服务部署在哪里
- 负责某个模块的开发者想知道模块内部怎么拆

于是就出现了两种常见错误：
1. 一张图试图塞所有层级 → 节点多到没法看，谁也看不懂
2. 没有图，只有文字描述 → 无法在脑子里建立空间感

C4 的核心洞察是：**架构理解本质上是一个"缩放"的过程，不同受众需要停在不同的缩放层级**。就像地图，看城市全局和看街道细节是两张完全不同的图，但它们描述的是同一个地方。

另外，C4 解决了一个"画了就过期"的问题：配合 Mermaid，图以文本形式存在仓库里，和代码一起做 code review，架构图的更新和代码的更新可以在同一个 PR 里完成。

### 什么时候用

- 向新人解释系统整体结构（L1/L2）
- 某个复杂模块拆分后，需要说清楚"谁依赖谁"（L3）
- 技术方案评审，需要结构化地沟通设计

### 什么时候不适合用

- 纯粹的数据流/时间顺序（用时序图）
- 代码级别的类图（L4，通常 IDE 自动生成更合适）
- 团队规模小、模块少（过度设计）

### 三个层级哪个最值钱

对大多数项目来说是 **L3（Component 图）**。L1/L2 相对稳定，L3 随着模块拆分快速变化，同时也最容易让人在脑子里迷失。

### 学习建议

1. **看 [c4model.com](https://c4model.com)** — 作者 Simon Brown 的官网，有 reference card 可以打印
2. **配合 Mermaid** — 语法见 [mermaid.js.org/syntax/c4](https://mermaid.js.org/syntax/c4.html)
3. **先画 L1 + L2** — 让读者先建立"前端/后端/数据库/外部服务"的边界感；某个模块复杂到目录结构已经无法表达依赖关系时，再补 L3

### 在 AI 编程中的价值与用法

**AI 最常犯的错误之一：错误的依赖关系**

当你要求 AI 添加新功能时，它经常会：调用一个不存在的模块（猜错了你的命名）、重新实现一个已有的功能（不知道某个 class 存在）、把代码放在错误的层（不知道业务逻辑该在 service 还是 route）、引入不符合项目运行环境的依赖（在 Cloudflare Workers 里引用 Node.js 专有模块）。

这些错误的根本原因：**AI 不知道你的项目当前的组件边界**，只能从你提供的文件内容猜测结构。

C4 的 L3 Component 图直接解决这个问题——它是一张"你的项目有什么组件、它们如何连接"的权威地图，AI 读到它，通常能降低错误依赖的概率。

**具体用法**

1. **把 L3 放进 CLAUDE.md**：让 AI 在每次会话开始时就能看到组件结构图。AI 在生成代码时会依据这张图决定"在哪个文件里写、依赖哪个现有组件"，而不是猜测。

2. **新功能前先确认落点**：添加新功能时，先提供 L3 图，问"这个功能应该放在哪个组件里？需要新增组件吗？"让 AI 在架构层先做决策，再进入代码实现。比事后发现代码放错地方再重构省得多。

3. **用图纠正 AI 的依赖错误**：AI 生成的代码引用了错误的依赖时，用图指出："根据这张 Component 图，X 不应该直接依赖 Y，它应该通过 Z。"有图的纠正比纯文字说明清晰，AI 也更容易按图修正。

4. **让 AI 更新图**：当新增了一个组件或改变了依赖关系，让 AI 同步更新 Mermaid 图。图住在仓库里，把"更新架构图"加进 PR checklist，就能保持图和代码同步。

5. **L1 帮 AI 理解技术栈约束**：L1 Context 图让 AI 看到"这个系统运行在 Cloudflare Workers 上，调用外部 LLM API"。AI 就不会建议用本地文件系统、长连接或 Node.js 专有模块。

**本项目的实际应用**

`Single-Agent设计文档.md` Section 4.1 是 AI chat 模块的 L3 图，明确了 8 个文件的职责和依赖方向。在修改这个模块时提供给 AI，它就知道"ToolExecutor 通过 ConflictDetector 处理冲突，不能直接绕过"这类架构约束，而不用你每次重新解释。

## 三、Diátaxis（文档四象限框架）

### 是什么

一个组织文档的框架，不是写文档的工具。它把文档分成四种类型，要求每种文档"保持纯粹"，不混杂：Tutorial（教程）、How-to Guide（操作指南）、Reference（参考手册）、Explanation（概念解释）。

### 为什么它存在（这是关键）

先问一个问题：**你的文档是写给谁看的？**

一个刚入门的用户，和一个有经验但需要查参数的用户，和一个想理解"为什么这样设计"的架构师——他们的需求是完全不同的，**但大多数文档只有一种形式，试图同时满足所有人**。

结果是：什么人都满足不了。初学者看不懂（充斥着假设读者已知的内容），老手查东西效率很低（概念解释混在参考手册里），架构师看不到设计的来龙去脉（淹没在操作步骤里）。

Diátaxis 的核心洞察来自学习科学：**人类在不同的阅读目标下，大脑工作模式不同**。

- 在"学习模式"时，你希望被引导，不需要也不应该做决策（Tutorial）
- 在"完成任务模式"时，你已经知道怎么做，只需要步骤提示（How-to）
- 在"查阅模式"时，你需要准确、完整的事实，不需要解释（Reference）
- 在"理解模式"时，你不急着做事，想建立更深的心智模型（Explanation）

把这四种混在一起，就像把食谱、烹饪原理、食材营养表、厨师故事全写在一页纸上——每个读者都要先在里面找自己需要的部分，认知负担极高。

**它解决的不是"写什么"，而是"把写好的东西放在哪里"**。

### 什么时候用

- 项目文档越来越多，但越来越难找到想要的
- 不同背景的读者（初学者 vs 老手）都觉得文档不好用
- 团队对"什么内容该写在哪里"没有共识

### 什么时候不适合用

- 文档量很少（3-5 个文件），分类带来的收益不抵得上维护成本
- 文档完全是内部自用，只有你一个人看

### 学习建议

1. **直接看 [diataxis.fr](https://diataxis.fr)** — 官网清晰、精炼，30 分钟读完核心内容
2. **先分类，再迁移** — 不要重写，先把现有文档贴标签（这是 Tutorial / How-to / Reference / Explanation 里的哪一类），自然就知道哪些混淆了
3. **从"Explanation 缺失"开始改** — 这类文档最常被忽略，但对新人和 AI 工具最有价值

### 在 AI 编程中的价值与用法

**文档类型直接影响 AI context 的质量**

AI 工具的 context window 有限，而且不是所有信息对 AI 等效有用。Diátaxis 的分类在这里有一个意外的价值：**四种类型的文档，对 AI 的用途截然不同**。

- **Explanation（概念解释）**：AI 最稀缺的 context。它告诉 AI 为什么这样设计——这正是 AI 最容易搞错的部分。AI 知道怎么写代码，但不知道你的项目为什么选了这个设计而不是那个。Explanation 文档填补了这个空白，让 AI 在边界情况下也能做出符合项目意图的决策。

- **Reference（参考手册）**：AI 最容易消费的 context。准确、完整、无歧义——这正是 AI 喜欢的格式。Tool definitions 表、枚举值清单、API 参数列表——这类文档 AI 能高效提取，不会误解。

- **Tutorial / How-to**：对 AI 的价值相对低，因为它们描述"人如何操作"，AI 是在代码层工作。但其中的"步骤"有时帮助 AI 理解项目的使用方式和前置假设。

- **混杂型文档（最常见的问题）**：AI 读到一个既有步骤又有解释又有参考数据的文档，提取效率大幅降低，甚至会把解释性文字当成规则来执行。

**具体用法**

1. **用 Diátaxis 思路组织 CLAUDE.md**：把 CLAUDE.md 内部分区——
   - **Reference 区**：硬性规则、禁止事项、枚举值（AI 必须遵守的事实）
   - **Explanation 区**：为什么有这些规则（帮助 AI 在规则没有覆盖的边界情况下推断意图）
   - **How-to 区**：常见任务的步骤（加新 route、加新 tool 等）
   
   这样 AI 读 CLAUDE.md 时，规则和背景是分离的，不会把解释性文字当规则，也不会忽略背景直接套规则。

2. **开始新会话时先提供 Explanation 文档**：每次开始一个新 AI 编程会话，优先提供 Explanation 类文档作为 context，而不是直接扔代码。"这是我们为什么这样设计 AI agent 架构"——AI 建立了这个背景，再读代码就事半功倍。

3. **查阅型任务用 Reference 文档**：当 AI 需要知道某个接口的准确格式、某个枚举的完整取值时，提供 Reference 类文档，而不是让 AI 从代码里猜。Reference 信息密度高，AI 读 10 行 Reference 比读 100 行混杂文档提取得更准。

4. **识别给 AI 的"噪音"**：如果你发现给 AI 的文档很长但 AI 总是忽略或误解某些部分，很可能是那份文档混合了多种类型。按 Diátaxis 拆分，只给 AI 当前任务需要的那种类型。

**本项目的实际应用**

| 文件 | Diátaxis 类型 | 给 AI 的时机 |
|------|--------------|-------------|
| `agent-design-principle.md` | Explanation | 修改 tool 定义或 prompt 结构前 |
| `Single-Agent设计文档.md` 架构图节 | Explanation | 添加新组件或重构模块前 |
| `tool-definitions.ts` 的描述 | Reference | 让 AI 理解现有 tool 规格时 |
| `AI-System-Prompt改进需求.md` 终稿节 | Reference | 让 AI 修改 system prompt 时 |

## 四、Living Spec（活文档 / 活规格）

### 是什么

把"规格说明"当作代码仓库里的第一等公民：它和代码住在一起，随 PR 更新，代码审查时一并审查，永远反映系统的当前真实状态。

### 为什么它存在（这是关键）

先想一个场景：你加入一个新团队，发现 wiki 里有一份"设计文档"，但它最后更新是两年前。你信不信它还准确？

大多数人不信。这不是团队懒，这是**经济学问题**：更新 wiki 的成本（切换上下文、找到文档、修改、发布）比收益（让某个未来的读者受益）高得多，所以理性选择是不更新。长期结果是，文档和代码越来越远，直到没人信文档，文档彻底失去价值。

Living Spec 的解法是：**把更新 spec 的成本降到接近零——因为它就在代码旁边，PR 自然顺带更新它**。

另一个更深的洞察：代码回答"是怎么做的"（How），但几乎不回答"为什么这样做"（Why）和"这个东西应该做什么"（What）。Living Spec 补上了代码无法自我描述的部分，而且因为住在仓库里，这部分描述会随代码一起演化。

### Living Spec 和 Spec Coding（AI 时代）的关系

这两个概念几乎是同一件事在不同时代的版本：

- **Living Spec（~2018）**：spec 给人类开发者读，通过流程保持和代码同步
- **Spec Coding（AI 时代，2023+）**：spec 同时给人和 AI 读；AI 读 spec 生成代码，AI 做决策时也更新 spec

你现在看到的 CLAUDE.md、`.cursorrules`、GitHub Copilot 的自定义指令文件，本质上都是面向 AI 的 living spec。AI 时代让 Living Spec 从小众实践变成了主流工作流，因为"AI 读的越清楚，代码质量越高"的激励让开发者真正愿意维护它了。

### 什么时候用

- 模块有复杂的业务规则，代码本身无法表达"为什么"
- 规则经常变化（AI prompt 行为约束、权限逻辑）
- 团队有 AI 辅助编码需求，需要给 AI 提供可靠的上下文

### 什么时候不适合用

- CRUD 量大、逻辑简单的模块（维护 spec 的收益不高）
- 团队还没有"PR 时一起更新文档"的文化基础

### 学习建议

1. **不要一次性写完整** — 从最复杂的一个流程开始，写一个 spec 文件，放进仓库，下次改代码时顺手更新
2. **spec 文件头注明"实现位置"和"最后同步日期"** — 便于判断是否已过期
3. **配合时序图效果最好** — 时序图描述"怎么流转"，living spec 描述"每一步的规则是什么"

### 在 AI 编程中的价值与用法

**Living Spec 就是 Spec Coding 的底层实践**

如果你听过"spec coding"或"spec-first development"，它描述的工作流是：先写清楚你想要什么（spec），再让 AI 实现。这和 Living Spec 的哲学完全一致——区别只是受众变了：传统 Living Spec 的读者是人类开发者，spec coding 里的读者是 AI。

你现在每天用到的 CLAUDE.md、`.cursorrules`、`.github/copilot-instructions.md`，本质上都是面向 AI 的 living spec。它们和代码住在一起，随项目演化，就是 Living Spec 理念的直接实现。

**为什么 AI 时代 Living Spec 变得更重要**

传统开发中，Living Spec 面临的最大阻力是："为什么要花时间写 spec？反正我自己知道要做什么。"这个阻力让大多数团队放弃了它。

AI 时代这个阻力消失了：**AI 不知道你想做什么，必须通过 spec 告诉它**。没有 spec，AI 只能猜测意图，这就是"vibe coding"——短期能出代码，但代码不符合你的架构约束、业务规则和设计意图，积累技术债，而且越来越难用 AI 维护。

Spec coding（= Living Spec 的 AI 版）是 vibe coding 的对立解法：**把你的意图写清楚，AI 在约束内实现，两者的边界清晰。**

**具体用法**

1. **把 CLAUDE.md 当成项目的行为规格文档，而不只是"给 AI 的提示"**  
   它应该包含：系统核心约束（什么不能做）、架构决策和原因（为什么这样组织）、常见模式（这个项目里某类问题的标准解法）、技术栈边界（允许用什么、禁止用什么）。  
   每次你发现 AI 做了一个你不想要的决定，问自己："这个约束在 CLAUDE.md 里写了吗？"如果没有，加进去。

2. **Spec-first 工作流**  
   添加新功能时，不要直接说"帮我实现 X"。而是先写一个 spec 描述：这个功能做什么、触发条件、边界情况、返回什么、不做什么。然后让 AI 对照 spec 实现。spec 文件留在仓库，下次修改时作为 context 提供。

3. **让 AI 帮你维护 spec**  
   每次 coding 会话结束后，问 AI："今天做的决定里，有哪些需要更新到 spec 文件？"让 AI 输出 spec 的更新建议，你 review 后 merge。这样维护成本会显著下降，而收益是下次会话时 AI 有更准确的 context。

4. **Spec 作为 code review 的基准**  
   在 PR review 时，把 spec 文件和代码变更一起看。如果代码改了但 spec 没改，要么是 spec 过期了（需要更新），要么是代码偏离了 spec（需要对齐）。这个步骤加进 PR checklist 后，spec 漂移问题自然消失。

5. **存量代码的 spec 补全**  
   让 AI 读现有代码，生成一份 spec 草稿："这段代码的行为规格是什么？"你 review 并修正。这是把隐性知识显式化的最低成本方式，之后 AI 就有这份 spec 作为可靠 context，而不是每次从代码重新推断。

**本项目的 Living Spec 实例**

| 文件 | 作为 living spec 的内容 | 修改时需提供给 AI |
|------|------------------------|----------------|
| `CLAUDE.md`（根目录）| 项目级开发约束 | 每次会话 |
| `docs/spec/create-task-flow.md`（建议新增）| create_task 完整行为规格、边界情况 | 修改创建任务相关逻辑时 |
| `docs/agent-design-principle.md` | AI agent 设计原则 | 修改 tool 定义或 agent 结构时 |
| `docs/agent-system-prompt-design-guide.md` | system prompt 写法准则 | 修改 prompt-builder 时 |

如果 `docs/spec/create-task-flow.md` 建立起来，下次修改 `ToolExecutor` 中 `create_task` 的逻辑时把它提供给 AI——它在这份 spec 的约束下修改代码，比直接说"帮我改这个函数"要稳定得多。

## 如何组合使用

这四种方法不是互斥的，组合使用效果最好：

```
时序图 + C4 L3  → 解决"我在哪里/整体怎么流转"的导航问题
Diátaxis        → 解决"我该去哪里找我需要的文档"的组织问题
Living Spec     → 解决"规则是什么/为什么"的内容问题
```

**最小可行组合（适合个人/小团队项目）：**
1. 一张系统/模块架构图：小项目优先 L1+L2，复杂模块再补 L3
2. 一张关键流程的时序图：说清楚主流程
3. 一个 living spec 文件：记录最复杂模块的行为规则

这三样东西加在一起，能让新人在一小时内建立足够的心智模型，也能让 AI 工具在生成代码时有足够的上下文。

## 一个根本性的转变

以前，写文档的动机是"为别人"——为以后接手的开发者，为新人，为做 code review 的同事。这个动机很弱，所以大多数团队的文档质量很低。

AI 时代，这个动机变了：**文档也是写给 AI 的**。你给 AI 的 context 质量，直接决定 AI 生成代码的质量。这四种方法在这个背景下的价值：

- **时序图**：把流程意图结构化地传给 AI，替代散文描述
- **C4 图**：把架构约束传给 AI，防止 AI 生成错误依赖
- **Diátaxis**：让你知道给 AI 提供哪种类型的文档，以及何时提供
- **Living Spec**：持续维护 AI 可信赖的 context，而不是每次会话都从零解释

这不是"文档变得更重要了"这种抽象结论。具体来说：在这四个方向投入时间，在文档准确且和当前任务匹配的前提下，通常能提升 AI 修改代码的稳定性，减少需要纠正的次数。回报率取决于文档是否保持和代码同步——过期的文档比没有文档更糟糕，因为它会以错误的信息误导 AI。
