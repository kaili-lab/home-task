## 1. 这个目录是什么，不是什么

**它是什么**：

- 针对你笔记里 6 个具体问题的完整回答
- 一份"现在的你"写给"三个月后的你"的参考笔记
- 重点锚定在 Home Task 这个具体项目 —— 例子、建议、优先级都基于你的真实代码

**它不是什么**：

- 不是一本系统的复杂度管理教材（真正系统的在下面的资源清单里）
- 不是 best practices 汇编（这类汇编通常把"原则"和"技巧"混在一起，学了技巧离开语境就废）
- 不是覆盖所有复杂度议题 —— 分布式、并发、性能等话题都没讨论，因为你当前的问题不在那里

**承认这套笔记的局限很重要**。不要把它当成完整的软件工程教程去读，那样会让你觉得"我学完了"，然后停下来。真正的态度应该是：**这是一组具体答案的索引，你将来每撞到一次复杂度墙，就回来翻对应的那一节。**

## 2. 怎么读这套笔记

基本顺序见 `00-index.md`。这里补充几条不在 00-index 里的读法建议：

- **第一次**：按 `01 → 02 → 03 → 04` 过一遍。目标是"知道有这些工具存在"，不求记住。第一次读完后你 **必然忘掉 70%**，这是正常的 —— 别因此怀疑自己的理解力。
- **动手之后**：当你在 Home Task 里撞到具体问题（"这个函数要不要拆"、"这条规则要不要抽出"），回来翻对应章节。**带着问题读比无目标读的效率高 10 倍**。
- **3 个月后**：整套再过一遍。你会发现很多当初觉得"显然"的话其实没读懂；也会发现一些当初觉得"抽象"的话变得具体。这一次的 ROI 通常比第一次高。
- **如果你只想先做一件事**：读 `01` 的第 2.3 节（Ousterhout 的三个观察指标），把 *change amplification / cognitive load / unknown unknowns* 装进脑子。这三个词是你将来评审任何代码、设计文档、AI 输出时最常用的"仪表"。

**什么时候 *不* 应该读这套笔记**：

- 当你还没有动手写过真实代码时 —— 理论读得再熟也不会变成判断力
- 当你想"学完再开始"时 —— 这是最常见的陷阱
- 当你希望它能给你"一套可复制的套路"时 —— 它不会

## 3. 想系统学习复杂度管理？外部资源清单

这套笔记是围绕你的问题展开的，不是系统教材。如果你想补系统性，按下面的顺序：

### 3.1 书（按推荐先后）

| 资源 | 篇幅 | 为什么读 |
| --- | --- | --- |
| **Ousterhout, _A Philosophy of Software Design_（2nd ed., 2021）** | ~190 页 | 我见过最接近"指南性"的复杂度管理书。每一条规则都配"撞到它时的感觉"。**第一本要读的** |
| **Brooks, _No Silver Bullet—Essence and Accident_（1986）** | ~20 页论文 | "本质 vs 偶然"的原始出处，网上免费，1 小时读完 |
| **Parnas, _On the Criteria for Decomposing Systems into Modules_（1972）** | ~10 页论文 | "信息隐藏"的原始提法。读起来有点年代感，但非常短 |
| **Evans, _Domain-Driven Design_（2003）** | ~560 页 | 厚，但前 3 章就能给你 bounded context 的核心想法。后面的章节等你真的在做大领域时再读 |
| **Chip Huyen, _AI Engineering_（2024）** | 一本实用书 | 不讲复杂度理论，但讲"你的 chat 模块所在的那一层"具体怎么做。AI 时代最有用的一本 |
| **Kent Beck, _Test-Driven Development by Example_（2002）** | ~240 页 | 对"测试即规格"的经典论述。在 AI 时代重新焕发意义 |

### 3.2 视频 / 博客 / 持续资源

| 资源 | 形式 | 为什么 |
| --- | --- | --- |
| **Rich Hickey, _Simple Made Easy_（2011）** | 45 分钟演讲视频 | Simple vs Easy 的分离。**建议反复看** |
| **Simon Willison 的博客**（simonwillison.net） | 持续更新 | AI 编码的长期实战观察。建议订阅 |
| **Thoughtworks Technology Radar** | 每半年更新 | 知道现在业界在往哪儿走 |
| **GitHub Spec Kit**（github.com/github/spec-kit） | 代码库 + 文档 | SDD 的参考实现。先读它的模板，暂时不用搬工具链 |

### 3.3 主动避免的东西

- **任何号称"AI 时代完整复杂度管理指南"的书**：这本书不存在。2026 年这个时间点，任何这样的承诺都是营销
- **"Best Practices" 类汇编**：这类书的问题是把原则和技巧混在一起，读者学到的是技巧，换到新语境就废了
- **只讲"应该"不讲"为什么"的书**：没有 why 的 rule 没法迁移，只能照抄，越照抄越偏

## 4. 怎么把这些概念真正内化？

这套笔记里的概念（领域 vs 架构选择 vs 实现偶然、深模块、单一事实源、三种真相、change amplification 等）你看一遍 **一定** 记不住 —— 别怀疑自己，没人能靠读一遍就记住抽象概念。真正内化需要一套具体的节奏。

### 4.1 五条对"读 + 实践 + 反思"的调整

**(1) "多读"常常被高估。** 同一本书读 3 遍不动手，第 3 遍的收益会很小。更有效的节奏是：**读一次 → 动手 → 回头重读刚才用到的那一节**。撞过墙之后再读，你会看到第一次完全看不到的东西。Ousterhout 的第 6 章我第一次读觉得是废话，写过两个浅模块之后再读，每一句都带刺。

**(2) "当前项目实践" 是性价比最高的一步。** 没有这一步，前面所有阅读都是装饰。你 **80% 的成长会来自这里**。你现在的 chat 模块正好是一个"恰好到撞墙"的完美练习场 —— 这种状态其实很稀缺，别等"准备好"再动手。

**(3) "新项目实践" 重要但不紧急。** 这一步是为了验证你学到的是"原则"而不是"技巧"（原则能迁移，技巧不能）。理想节奏是 **6 个月一个新小项目**，刻意用一次这套方法论。

**(4) "反思总结" 关键，但有天花板 —— 不要过度工程化这个元过程。** 最便宜又最有效的反思是：每次改完一个 bug 或做完一个重构，花 5 分钟问自己 §5.1 的那两个问题。

**(5) 最容易漏的一步：写出来讲给另一个人听。** 这是把"我好像懂了"变成"我真的懂了"最快的方法。形式不重要 —— 博客、写给自己的设计文档、跟朋友 30 分钟分享、甚至是你让 AI 帮你写这套文档本身 —— 都算。**"把它放进词里" 的动作本身会暴露你以为懂但其实没懂的地方。**

### 4.2 最大的风险

**过度琢磨"学习方法"而不是直接动手。** 理论是便宜的，上限很低。真正贵的是实践 + 犯错 + 改。那几本书加起来可能花你 40 小时；把一条原则真正用在 Home Task 上可能花你 40 小时。**这两份 40 小时的价值完全不对等 —— 后者高出一个数量级。**

所以你读这份 README 的时候，最危险的做法是："好，我先把这几本书都读完再动手"。正确的做法是："我现在就开始 `02` 的优先级 1（写 ADR-0001），边做边翻 Ousterhout 对应章节。"

### 4.3 一个具体的 6 个月节奏建议

- **这周**：读 Ousterhout 前 7 章（约 100 页）。跳过你觉得显然的部分
- **接下来 2–3 周**：动手做 `02-applied-to-chat-module.md` 的优先级 1（写 ADR-0001 钉住主线）+ 优先级 2（time-driven tests）。做完各写 100 字反思
- **1 个月后**：读完 Ousterhout。回头重读 `02`，你会看到现在看不到的东西
- **3 个月后**：开一个新小侧项目，刻意用一次这套方法论（定位一次三层复杂度、写一条 ADR、做一次 table-driven 测试即可）
- **6 个月后**：这整套文档再读一遍。**如果还觉得新鲜，说明你没在实践**；如果觉得"这些我都会了"，说明该升级到下一层东西

### 4.4 怎么知道你真的学会了？

几个可观察的信号 —— 你不是在做检查清单，而是这些信号会自然出现：

- 你读别人的代码时，会不自觉地在脑子里贴标签（"这是浅模块"、"这是 change amplification"、"这是 cognitive load 过载"）
- 你在写需求描述时，会本能地先区分"这是领域约束"和"这是实现选择"
- 你看到一条规则时，会自动问"这条规则的单一事实源在哪？"
- 你写代码前会先问"我能不能把这个决定推迟、或者藏起来？"
- 你在给 AI 写 prompt 时，会主动先想"我要约束它的是哪一层的复杂度？"

这些信号 **不是**：能背出 Ousterhout 的三个指标、能讲清 Brooks 的本质 vs 偶然、能写出 deep module 的定义。那些是表面功夫，随时可以 google 到。真正的内化是 **下意识的分类动作**。

## 5. 两张"做事时打开"的实用清单 + 一张速查表

### 5.1 修完一个 bug / 做完一个重构之后问的两个问题

花 5 分钟把答案写在 commit message 或一个简单的 `reflections.md` 里。不用长，2 行就够。

1. **这个问题属于三层分类里的哪一层？**
   - 领域复杂度？（必然存在的，只能建模）
   - 架构选择复杂度？（当前技术路线引入的，理论上可以绕掉）
   - 实现偶然复杂度？（写法导致的，可以直接重构）

2. **我刚才用到的是哪一条原则？如果我不知道这条原则，我会怎么做？**
   - 如果答案是"我根本想不出来"，说明你刚学到一个新东西，值得记下
   - 如果答案是"我会用另一种方式"，比较两种方式的代价

**这两个问题比任何复杂的反思模板都更能把隐性知识显性化。**

### 5.2 评审代码（自己的或 AI 的）时扫一遍的红旗清单

来自 Ousterhout 的"design red flags"。看代码时扫过这些问题 —— 任何一个"是"都值得停下来想想：

- [ ] **Change amplification**：改一个概念上的小东西，要同步动好几处代码吗？
- [ ] **Cognitive load**：为了安全地改这段，我需要在脑子里装多少上下文？
- [ ] **Unknown unknowns**：这段代码里有没有我没意识到的隐藏规则？（**最危险的一种**）
- [ ] **浅模块**：这个函数/类的接口复杂度和实现复杂度差不多吗？是不是只是搬了一层？
- [ ] **Simple vs easy 搞混了**：这个"方便"的选择是不是偷偷引入了"缠绕"？
- [ ] **信息隐藏失败**：这个模块暴露出来的信息，是不是本来可以藏起来的？
- [ ] **变化的轴没对齐模块边界**：如果这类需求将来变了，是不是要动多个模块？

**AI 协作特别提醒**：AI 特别擅长写"看起来对但暗藏陷阱"的代码（变量名合理、结构合理、甚至注释都合理）。对 AI 输出多用一次红旗清单，收益比对人类代码高。

### 5.3 开始一个新 feature 之前的 pre-mortem

在你让 AI 写代码 **之前**（或者自己动手之前），花 3 分钟问：

1. 这个 feature 主要加的是哪一层的复杂度？
2. 其中有哪些是领域本身要求的？哪些只是我的当前技术路线引入的？
3. 有没有办法把"架构选择复杂度"那部分砍掉、缩小、或者降级？
4. 如果这个规则 / 约束以后要变，我现在打算把它藏在哪里？

这一步花费极低，但能帮你避免 80% 的事后重构。它是 `04-ai-era-and-specs.md` 里"把思考时间从 IDE 里移到 IDE 外"的具体操作。

### 5.4 关键词速查表（可以贴在显示器边）

| 词 | 一句话解释 |
| --- | --- |
| **Essential vs accidental complexity**（Brooks, 1986） | 前者来自问题本身，不能消；后者来自你的实现方式，可以消 |
| **三层复杂度分类**（本系列 `02`） | 领域 / 架构选择 / 实现偶然。比 Brooks 的二分更实用，尤其对 AI 路线 |
| **Information hiding**（Parnas, 1972） | 模块边界应该画在"可能变化的维度"上，不是画在"计算步骤"上 |
| **Deep module**（Ousterhout, 2018） | 小接口、藏大量功能。反面是"浅模块"——搬一层代码没有净收益 |
| **Simple vs easy**（Hickey, 2011） | Simple = 不缠绕（客观性质）；easy = 熟悉方便（主观感受）。**追 simple，不追 easy** |
| **Change amplification**（Ousterhout） | 改一个概念上的小东西要同步动多处 = 红旗 |
| **Cognitive load**（Ousterhout） | 为了安全地做一个改动要背多少上下文 |
| **Unknown unknowns**（Ousterhout） | 你不知道自己不知道 —— 最危险的一种复杂度 |
| **Single source of truth** | 每条事实只写一次，其他地方都是派生或引用 |
| **三种真相**（本系列 `03`） | 期望真相（PRD/ADR）/ 实现真相（代码/测试）/ 运行真相（日志）。冲突时先找锚点 |
| **ADR**（Nygard, 2011） | Architecture Decision Record。追加写、记 Context / Decision / Consequences |
| **C4 Model**（Brown） | Context → Container → Component → Code 四层缩放图 |
| **Living documentation** | 文档和代码从同一个源头派生，不是手动同步 |
| **Bounded context**（Evans, 2003） | 领域里一块语言自洽、模型自洽的区域 |
| **SDD（Spec-Driven Development）** | 把规格（而不是代码）当作源头。2026 仍在升温，不是稳定共识，不是银弹 |

## 6. 在 AI 协作场景里怎么用这些

这是整套笔记的 **元主题** —— 既然你和 AI 一起写代码，上面这些概念怎么对 AI 生效？

### 6.1 让 AI 写代码之前

- **按三层复杂度把需求过一遍**：哪些是领域约束（AI 必须守住）？哪些是架构选择（你需要给方向）？哪些是实现选择（可以交给 AI 发挥）？这一步会立刻暴露你的需求描述里模糊的地方
- **带着规则表一起发给 AI**：如果某条规则是领域复杂度，不要只说"按规则来"，而要把规则原文贴上去。这就是为什么 `02` 优先级 2 推荐把时间规则做成 table-driven tests —— 这张表可以直接作为 AI 的上下文
- **先告诉 AI 你的主线**：例如"我们现在主线是单 agent，不要引入 LangGraph"。这能让 AI 的输出不会突然变成另一种架构

### 6.2 评审 AI 输出时

- **用 §5.2 的红旗清单扫一遍**，尤其是 unknown unknowns 和 change amplification
- **问一次**："这是深模块还是浅模块？"、"这段引入了 change amplification 吗？"、"AI 写的测试到底在测什么 —— 行为还是实现？"
- **不要只看它是否跑通**：跑通只是下限。真正的评审问 "它未来能不能改"

### 6.3 当 AI 的输出和你的预期不符

- **先问自己**："是我没讲清楚，还是 AI 理解错了？" —— 大多数情况是第一个
- 如果是第一个，**修改你的需求描述**，而不是让 AI 无尽地改代码
- 提示词工程的真实含义：**把一个需求讲到让另一个智能体（哪怕是 AI）不需要猜就能实现**。这和过去"给队友写一份能照做的任务说明"完全一样

### 6.4 一个值得保留的习惯：多模型交叉审核

你写这套笔记时做过一件事：让 Claude 写 → 让另一个模型（GPT）审核 → 把审核拿回来让 Claude 应用修改。**这是一个非常值得保留的习惯**，理由：

- **不同模型的盲点不同**：Claude 可能过度接受某些假设，GPT 可能更容易挑出宏观错误；反之亦然
- **审核比自审更可靠**：一个模型写出的内容，让另一个模型审查，比让同一个模型自己审查更能发现问题
- **你在这个循环里是决定者**：不是"哪个模型写得对就听哪个"，而是"两个不同视角帮你看到你自己看不到的部分"

这个习惯在 AI 时代相当于传统的 code review，只是 reviewer 不是人类队友而是另一个模型。**独立开发者最缺的就是"第二双眼睛"**，这个办法最接近补上这个缺口。

**注意**：不要把这变成"让三个模型投票"。三个模型的共识不代表正确，只代表它们共享了相同的训练数据偏差。真正的价值 **在于看到分歧** —— 两个模型意见不同的地方，就是值得你自己深思的地方。
