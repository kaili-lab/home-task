# 03 · 文档也是工程：PRD 与模块文档的一致性问题（Q3 + Q4）

> 回答你笔记里的两个问题：
> - Q3：怎么在全局 PRD 和模块内部文档之间建立"联系"，让需求变动能同时反映在两边？
> - Q4：代码是肉体、文档是精神 —— 这个比喻对吗？

## 0. 先说结论

- **Q3 的短答案**：不要追求"让所有文档互相同步"，而是要做到"**同一个事实只写在一个地方，其他地方只做引用**"。同步问题只能通过消灭重复来解决，而不能靠纪律。
- **Q4 的短答案**：你的比喻抓到了一个真问题，但方向反了 —— 在软件里，**代码才是"现在正在发生的事"，文档是我们对它的地图**。更准确的比喻是 **"地图与疆域" (Korzybski)**，下面会展开。
- **最容易做错的事**：写了很多文档，半年后它们和代码完全不一致，于是新来的人不敢信文档，也不敢改文档 —— 你同时付出了"写"和"不敢信"两份成本。避免这一点，比写得漂亮更重要。

---

## 1. 为什么你觉得 "PRD 和模块文档的联系" 很难建？

先定位问题。你笔记里说的 "怎么建立联系" 其实是两个子问题纠缠在一起：

1. **结构联系**：PRD 说"系统要支持创建任务"，模块文档说"`create_task` tool 有这些参数" —— 读 PRD 的人怎么找到模块文档？读模块文档的人怎么回到 PRD？
2. **语义一致性**：PRD 说"用户未提供日期默认今天"，代码里也有这条规则，文档里也有这条规则 —— 三处同步怎么保证？

这两个问题的解法是不一样的。结构联系靠"组织方式 + 引用"；语义一致性靠"单一事实源 + 生成"。

### 不要犯的第一个错：把 PRD 写成规格手册

很多人（包括我刚开始的时候）会把 PRD 扩成一本包含所有细节的"规格手册"，然后尝试手动保持它和代码一致。这是 **注定失败的**，因为：

- PRD 的读者是"想了解这个产品在做什么"的人（你未来的自己、潜在合作者、面试官）。他们需要的是 **鸟瞰视图**，不是细节。
- 细节必然漂移。每次改代码都更新 PRD 细节是不可持续的纪律负担。
- 细节一旦过期，整个 PRD 的可信度就崩了。读者会开始"不相信 PRD"，然后这份文档就成了僵尸。

**正确做法**：PRD 只包含 *相对稳定* 的内容。

- 产品是做什么的（问题 / 用户 / 价值主张）
- 核心概念（任务、群组、时间段、提醒 —— 定义而非实现）
- 功能清单（用户能做什么，不说怎么做）
- 跨模块的不变量（例如"所有写操作必须鉴权"）

*不* 放：具体 API 签名、具体实现细节、可能下周就改的规则参数、具体 LLM prompt 内容。

## 2. 文档的五种"物种"——不要混着养

把"代码外的东西"一概叫"文档"，就像把所有动物都叫"动物"一样粗糙。它们各自有不同的生存规律和维护成本。

| 类型 | 回答的问题 | 例子（你项目里的） | 变化频率 | 维护策略 |
|---|---|---|---|---|
| **PRD / 产品规格** | 这个系统对外做什么？ | `docs/原始PRD.md` | 低 | 只写稳定的、从用户视角出发的内容 |
| **架构 / 设计文档** | 这个系统内部怎么组织？ | `docs/Single-Agent设计文档.md`、`docs/multi-agent-design.md` | 中 | C4 多层级；配 Mermaid 图 |
| **ADR (决定记录)** | 为什么我们做了这个选择？ | （还没有，建议建立） | 极低（append-only） | 追加写，永不修改 |
| **Rule Book (规则手册)** | 领域里的不变量是什么？ | 你目前零散在 PRD §"重要更新(时间段逻辑)" 和代码里 | 中 | 单一事实源；从它生成 prompt、代码常量、测试 |
| **Runbook / 操作手册** | 出问题时怎么处理？ | （小项目通常不需要） | 中 | 问题发生后再写 |

**每种文档有不同的变化节奏。** PRD 几乎不动；ADR 永不动；设计文档随大重构动一次；Rule Book 随业务规则动。把它们混在一个 `PRD.md` 里，你就把五种节奏绑到一起，任何一个变化都要动这一个文件。那个文件就必然会烂。

**最常见的混养错误**：

- 把 Rule Book 塞进 PRD（→ PRD 必须频繁更新）
- 把 ADR 塞进设计文档（→ 设计文档无法干净重写）
- 把 Runbook 塞进 README（→ README 臃肿、读者迷路）

分开。每种文档一个目录或一个明确位置。

## 3. 怎么建立"联系" —— 三种机制

### 机制 1 — 单一事实源 + 引用

这是最重要的一条。**每一条事实只住在一个文件里。** 其他地方需要引用它时，通过链接或者程序化生成。

**反例**（当前你项目里的情况）：

"今天未到晚上时默认 all_day；今天已到晚上时默认 evening" 这条规则现在同时出现在：
- `原始PRD.md`
- `docs/Single-Agent设计文档.md` §5
- `docs/Single-Agent设计文档.md` §7
- `prompt-builder.ts` 的 `buildSystemPrompt()`
- `prompt-builder.ts` 的 `getDefaultTimeSegmentForDate()`

**改成一个 Rule Book**（见 `02-applied-to-chat-module.md` §3 优先级 1）：
- 规则只写在 `rules/time-segment-rules.ts` 的常量里
- Prompt 从它生成
- 代码从它读取
- 文档从它生成 Markdown 表格
- 所有 PRD / 设计文档提到这条规则时，只写：**"详见 [时间段规则](../src/services/ai/rules/time-segment-rules.ts)"**

规则变了，只改一个地方。**同步问题被从根上消灭了，而不是靠纪律维持。**

### 机制 2 — 双向引用 (Bidirectional Links)

对 *结构联系*（"读 PRD 的人怎么找到模块文档"），建议在两边都放指针：

- 在 `原始PRD.md` 的"AI 对话功能"一节末尾写一行：**"实现细节见 [`docs/Single-Agent设计文档.md`](./Single-Agent设计文档.md)**"
- 在 `Single-Agent设计文档.md` 开头写一行：**"本文档是 [原始PRD §6.X AI 对话] 的实现设计"**

这样从任一侧都能跳到另一侧。对读者来说是极大的体验提升，对你来说维护成本几乎为零。

### 机制 3 — PR 纪律（cheap，但有效）

当你改一个模块的核心文件时，问自己三个问题（写在 PR 描述里）：

- [ ] 这个改动有没有违反 PRD 里的某条承诺？
- [ ] 这个改动有没有让某条 ADR 变得不再成立？如果有，是不是要写一条新的 ADR 覆盖它？
- [ ] 这个改动有没有引入新的 Rule Book 条目，或者让旧条目过期？

这三条不需要工具，只需要自己在提交前过一遍。对一个独立开发者，PR 本质是"过去两小时的我"给"现在的我"写的消息 —— 这个 checklist 让你被迫回忆上下文。

### 什么时候上工具（先别上）

等你已经用单一事实源 + 双向引用走过 2–3 个月并且仍然觉得不够，再考虑工具：

- **Docusaurus / Astro Starlight**：把 `docs/` 站点化，自动生成侧边栏、跳转、搜索。
- **C4-PlantUML / Structurizr**：从一份源文件生成不同缩放级别的 C4 图。
- **`arc42` 模板**：一个成熟的架构文档骨架，结构比你自己发明的更标准。
- **`Wiki-like` MkDocs**：如果你决定 PRD 和设计文档都用 Markdown 并希望有漂亮的渲染。

**不要现在就上工具。** 工具是在"你已经有了清楚的内容组织方式"之后才有价值的 —— 否则只是把混乱搬到另一个地方。

## 4. 关于"代码是肉体、文档是精神"的比喻

你的比喻有一个非常真实的内核：**文档承载了代码本身无法表达的东西 —— 意图、理由、曾经的权衡、被放弃的方案。** 这些都不会被编译器检查，不会在运行时被执行，但它们是让代码可以被理解、被继续演化的关键。这一层的存在是真的，你感觉到了它。

但比喻有一个方向是反的，会把人带偏。

### 4.1 为什么 "代码是肉体、文档是精神" 在软件里是危险的比喻

这个比喻来自"思想比物质更根本"的哲学传统（柏拉图、笛卡尔）。在那个传统里：精神/思想是第一性的、不朽的；肉体是第二性的、临时的。

但在软件里，**事实恰好反过来**：

- **代码是 "正在真实发生的事"**。它在服务器上运行，它处理用户的请求，它改变数据库。它是事实。
- **文档是我们对代码的 *叙述***。叙述可能准、可能不准，可能新、可能过期。叙述本身不执行任何东西。

如果你按"文档是精神、代码是肉体"的比喻去工作，你会下意识地认为：
- 文档是更根本、更可信的
- 文档和代码不一致时，应该"把代码修得跟文档一致"
- 文档在某种意义上"先于"代码

**这些直觉在很多时候是错的。** 现实中：
- 代码每天都在演化，文档很少跟上
- 文档和代码不一致时，十有八九是文档错了
- 用户看到的是代码的行为，不是文档的承诺

所以这个比喻会让你过度相信一份你其实已经很久没看过的文档。

### 4.2 一个更有用的比喻："地图与疆域"

Alfred Korzybski 在 1931 年说过一句很出名的话："**地图不是疆域** (The map is not the territory)。"

- **疆域 = 代码**：它是"真的"，它在运行，它在变化。
- **地图 = 文档**：它是对疆域的表示。
- 好的地图 **帮助你在疆域里导航**，但它永远只是简化。
- 地图 **必须被更新**，否则它会逐渐和疆域脱节。
- 当地图和疆域不一致时，**先问"哪一份才是期望真相的锚"**（见下面 §4.3 的展开）。找到锚以后再判断要不要更新地图。

这个比喻对你当前的问题更有用，因为它立刻回答了几个疑问：

| 你的疑问 | 地图比喻给出的答案 |
|---|---|
| PRD 和模块文档怎么联系？ | 像一份国家地图和城市地图 —— 缩放级别不同，都指向同一个疆域 (代码)。它们之间不需要直接同步，它们都只需要和疆域保持准确。|
| 一份文档应该多细？ | 跟它的缩放级别匹配。PRD 是国家地图 —— 不画小巷。模块设计文档是城市地图 —— 不画室内装修。|
| 代码变了，文档要不要跟着变？ | 先问：地图还准吗？不准了才更新。但要承认地图总是稍微滞后的，这没问题。|
| 一个决定为什么要记到 ADR 里？ | 地图只告诉你"这里有一条路"；ADR 告诉你"为什么当初选了这条路而不是另一条"。这是地图本身没法表达的知识。|

还有一个类似的、更针对软件的比喻来自 Pieter Hintjens："**代码是被冻结的决定 (code is frozen decisions)**"。文档和 ADR 的工作就是记录被冻结前的那段推理。

### 4.3 但"相信疆域"不能机械执行 —— 三种真相

上面这个比喻有一个陷阱（感谢一次 GPT 交叉审核帮我把这一点收紧了）：如果你把"地图和疆域冲突时相信疆域"直接当成工程规则，很容易滑到一个危险习惯：

- 代码现在这么跑，那就默认它是对的
- 文档冲突了，多半是文档过期
- 那就去改文档去适配代码

但真实工程里并不是只有一种"真相"。至少存在三种：

| 哪种真相 | 它说什么 | 存在于哪里 |
| --- | --- | --- |
| **期望真相** | 产品 / 规则 / 架构 *本来想要* 什么 | PRD、ADR、Rule Book、原始需求对话 |
| **实现真相** | 代码和测试 *现在写成* 了什么 | 运行的源代码、已存在的测试 |
| **运行真相** | 线上实际 *表现* 成什么 | 日志、监控、真实用户行为 |

这三种真相 **在理想状态下应该一致**，但在真实工程里经常不一致。当你发现两份文件冲突时，正确的问法不是"信谁"，而是：

1. **是实现真相偏离了期望真相？** —— 代码在重构时跑偏了，ADR / PRD 还是对的 → 应该修代码。
2. **是期望真相变了、但没更新文档？** —— 新需求落地了，但 PRD 没跟上 → 应该更新 PRD。
3. **是期望真相本身就该重写？** —— 原始设计考虑不周，现在的代码其实是"对的" → 应该写新的 ADR 覆盖原 ADR（Supersedes）。
4. **是运行真相暴露了两边都不对？** —— 线上数据显示你的规则在某个边界崩了 → 两边都要改。

**具体到你的时间规则**：它散落在 `prompt-builder.ts`、`agent-loop.ts`、`tool-executor.ts`、`原始PRD.md`、`Single-Agent设计文档.md` 里。当这 5 处不一致时（而你现在几乎一定是不一致的），不能简单说"以代码为准"。更稳的做法是：

- **先找期望真相的锚点**：5 份里哪一份才是"当初我想要这条规则干什么"的最权威表述？如果没有明确锚点，说明这条规则从来没有被正式固定下来 —— 这本身就是一个发现。
- **用 ADR 把锚点固定下来**，以后再有冲突都指向这份 ADR。
- **用 test 把锚点变成可执行的**（这正是为什么 `02-applied-to-chat-module.md` §3 里把 ADR-0001 和 table-driven tests 放到前两位）。
- **只有在锚点被明确的前提下**，"地图和疆域不一致时修地图"才是安全的默认动作。

所以上一节里那条"当地图和疆域不一致时，相信疆域"应当收紧为：

> **当地图和疆域不一致时，先问"哪一份才是期望真相的锚"。找到锚之后，检查代码是不是偏离了锚、还是锚已经变了、还是锚本身需要重写。机械地"以代码为准"会把 bug 合法化。**

地图/疆域的比喻本身没错 —— 它仍然比"精神/肉体"更适合软件。只是"信谁"这一步的判断，比原文写的要多一道手续。

### 4.4 你的直觉里对的部分

就算我说"比喻方向反了"，也不意味着你的直觉全错。你真正感觉到的是：

- **代码不能表达意图**。函数能告诉你它做了什么，不能告诉你你 *本来* 想做什么（以及为什么没选另一条路）。
- **代码不能表达被放弃的方案**。只有 ADR 能。
- **代码不能表达跨模块的约束**。这需要 PRD 或架构文档。
- **代码在细节层面总是正确的，但在"整体在做什么"层面可能让人迷路。**

所以文档的真正价值不是"和代码同步记录一次"，而是 **"表达代码永远无法表达的那一层 —— 意图、理由、放弃的方案、跨模块的约束"**。

当你这样想时，"文档是代码的地图"、"ADR 是代码的传记"、"PRD 是代码的使命宣言" —— 这些比喻都比"文档是精神"更可行。

## 5. 给你项目的具体建议（可以直接开始做）

1. **把 `docs/` 整理成三层**：
   ```
   docs/
   ├── prd/
   │   └── 原始PRD.md            ← 只保留稳定内容
   ├── design/
   │   ├── Single-Agent设计文档.md
   │   ├── multi-agent-design.md
   │   └── agent-design-principle.md
   ├── adr/                     ← 新建
   │   ├── 0001-manual-tool-loop.md
   │   ├── 0002-dual-agent-coexistence.md
   │   └── ...
   └── rules/                   ← 新建（短期）
       ├── time-segment-rules.md
       └── conflict-detection-rules.md
   ```

2. **在 PRD 顶部和每个设计文档顶部加"导航块"**：
   ```markdown
   > **相关文档**：[PRD](./../prd/原始PRD.md) · [ADR 列表](./../adr/) · [时间规则](./../rules/time-segment-rules.md)
   ```

3. **建一个 `docs/README.md` 作为 landing page**：列出上面三层的入口和每份文档的一句话说明。

4. **Rule Book 长期要搬成代码常量**（见 `02-applied-to-chat-module.md` §3 优先级 1），但是 *先写 Markdown 版本* 就行，等你确实需要从它生成 prompt 时再升级成 TypeScript 常量。"过早工程化" 本身是一种复杂度。

5. **写前 3 条 ADR**。哪怕每条只有 10 行。写的行为本身就会帮你把想法澄清。

## 6. 一个可复用的原则

如果你只记住这篇里一句话，记这句：

> **不要追求"文档和代码同步"。追求"每个事实只活在一个地方"。同步问题只在有多份副本时才存在 —— 没有副本，就没有同步问题。**

这就是为什么前面反复强调"单一事实源"、反复强调"Rule Book"、反复强调"生成而不是复制"。这些都是同一个原则的不同应用。
