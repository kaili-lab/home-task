# 01 · 软件工程复杂度到底指什么？（Q1 + Q2a）

> 回答你笔记里的两个问题：
>
> - Q1：软件工程是不是主要就是在解决复杂度问题？
> - Q2a：业界管理软件工程复杂度的通用思路有哪些？

## TL;DR

你的直觉是对的。把"软件工程"这门学科最干净的描述之一就是：**在需求不断变化、参与的人也不断换的前提下，长期管理系统的复杂度。** 这是学界和工业界的共识。从 Brooks（1986）到 Parnas（1972）、Ousterhout（2018）、Hickey（2011），所有经典文献都同意这一点。分歧只是在于 _怎么定义复杂度_ 以及 _推荐哪些技术去对抗它_。

本文给你一份地图。后面几篇文档都会引用这里的概念，所以建议先读这一篇。

---

## 1. 为什么"复杂度"是对的关键词

一个更中性的说法可能是："软件工程是在造能跑的系统。" 但这个说法解释不了软件工程为什么 _难_。单独写一个小程序让它跑起来并不难，很多没学过 CS 的人都能做到。真正难的是：

1. 需求变了之后这东西还能跑。
2. 换个人（包括三个月后的你自己）还能看懂、能改。
3. 跟别的系统对接时不会冒出奇怪的副作用。

复杂度是这三件事共同的敌人。你自己的描述其实已经非常精准了："chat 模块的复杂度已经超出我能掌控的范围。" 这种 _"我没办法把整个系统装在脑子里"_ 的感觉，就是复杂度撞墙的定义。

## 2. 四位经典作者怎么看复杂度

这四本（或四篇）材料是这方面最有用的源头，我强烈建议你至少读 Ousterhout 的书（最短最现代，约 180 页）。

### 2.1 Fred Brooks — 《No Silver Bullet》（1986）

Brooks 第一次把复杂度分成两类：

- **本质复杂度 (essential complexity)**：问题本身自带的复杂度。你在做一个时间助手，要处理时区、重复任务、冲突、"已经过去的时段"这种规则，还有"下午"这种模糊时间表达 —— 这些复杂度是 _问题_ 带来的，不是你写代码的方式带来的。你消不掉。
- **偶然复杂度 (accidental complexity)**：你选择的实现方式带来的复杂度。糟糕的抽象、别扭的工具、重复的代码、纠缠的状态、职责不清。这些 _可以_ 被降低或消除。

Brooks 的悲观结论：本质复杂度占大头，所以没有任何单一的工具或语言能给你 10 倍的生产力提升。
Brooks 的乐观推论：**尽可能把所有偶然复杂度都砍掉，因为那才是你能赢的地方。**

这个分类对你很有用。当你觉得 chat 模块"太复杂"时，第一个要问的问题就是：_这是本质的还是偶然的？_

- "判断时间段是否已经过去" → 本质。任务管理助手必须处理这个。
- "LLM 可能骗你说已经创建了任务但其实没调工具" → 本质（只要你用 LLM，这个幻觉问题就存在）。
- "AgentLoop 里的 10 轮循环 + 跨轮 state tracking + 多个 early-return 路径缠在一起" → 大概率偶然。可以重构。

### 2.2 David Parnas — 《On the Criteria for Decomposing Systems into Modules》（1972）

Parnas 的核心洞察是：模块的边界不应该按照 _计算的步骤_ 切，而应该按照 _可能会变化的维度_ 切。他管这个叫 **信息隐藏 (information hiding)**。

每个模块把一个"设计决定"藏在稳定的接口后面。等到这个决定需要改的时候，只有这一个模块要改。

这是所有"模块化"箴言的祖宗 —— 封装、单一职责、DDD 的 bounded context、微服务，都是从这里长出来的。

Parnas 留给你的一句箴言：**模块边界应该画在"可能变化的轴"上，而不是画在名词或阶段上。**

举例：你的 chat 模块现在按"阶段"分 —— `prompt-builder` → `agent-loop` → `tool-executor`。这是按 _步骤_ 分的。如果改成按"可能变化的维度"分，会更像：

- 时间规则（可能整个替换）
- 冲突检测策略（可能加新的算法）
- LLM 适配层（可能换模型）
- Tool 定义（会经常改）

你其实已经在往这个方向走，只是还没完全到位。

### 2.3 John Ousterhout — 《A Philosophy of Software Design》（2018）

Ousterhout 给了我见过最实用的 _定义_：

> 复杂度就是任何让系统难以理解、难以修改的东西。

他列了三个可观察的症状：

1. **Change amplification（改动放大）**：一个概念上很小的改动，在代码里却要在很多地方同步改。典型信号：你改了一个规则，结果需要同时改 prompt、改 tool 描述、改测试用例、改前端兜底逻辑。
2. **Cognitive load（认知负担）**：为了安全地做一个改动，你需要在脑子里装多少东西。
3. **Unknown unknowns（未知的未知）**：你甚至不知道自己不知道什么。看代码看不出你需要注意哪些隐藏规则。这是最糟的一种 —— 你连要问的问题都问不出来。

> 对你当前的项目来说，最危险的是第三种。比如：新来一个人（或者半年后的你自己）看 `AgentLoop.chat()`，根本猜不出"今天 + 时段已过"这条短路规则为什么要在 LLM 循环之前就判断 —— 而且一旦你忘了它，改 prompt 的时候就会在某个分支留下 bug。

Ousterhout 最核心的建议是 **深模块 (deep modules)**：模块应该有小而简单的接口，但藏着大量功能。

- 浅模块：接口几乎和实现一样复杂 —— 只是搬了一层代码，没有净收益。
- 深模块：调用方用很小的表面积就能完成强大的事。

你代码里的 `ConflictDetector` 正好是这个思路的好例子。调用方只说 `findSemanticConflicts(tasks, title)` —— 一个三个参数的接口，背后藏了标题归一化、bigram 生成、Dice 系数比对、阈值判断。这就是一个深模块。值得表扬。

反面例子在同一份代码里也能找到：`HallucinationGuard` 有一堆彼此独立的小方法（`inferTaskIntent` / `shouldRequireToolCall` / `shouldSkipSemanticConflictCheck` / `looksLikeActionSuccess` / `buildActionNotExecutedMessage`），每个都很浅，调用方必须知道在什么时机用哪个。这就是一个偏浅的模块。不是灾难，但是有改进空间。

### 2.4 Rich Hickey — 《Simple Made Easy》（2011）

Hickey 把两个我们常混用的词分开了：

- **Simple**（拉丁语 _sim-plex_，"一折"）= 不缠绕、不和别的东西混在一起。**这是一个客观性质。**
- **Easy**（拉丁语 _adjacens_，"近在手边"）= 熟悉的、方便的、跟你已经知道的东西靠得近。**这是一个主观感受。**

一个 monkey-patching 的框架可以是 _easy_（少打字）但同时是 _complex_（它把你的代码和框架生命周期编织到一起，两边都更难独立推理）。

Hickey 的核心主张：**追 simple，不要追 easy。** Easy 只能买到短期速度，simple 才能买到长期灵活性。

对你的实际影响：当你在 chat 模块里纠结"要不要把 X 规则塞进 prompt 让 LLM 去判断" vs "要不要在 Tool 代码里硬编码 X 规则"时 —— 塞进 prompt 往往 _easy_（不用多写代码），但会让整个系统变 _complex_（规则分散在 prompt 和代码里，缠在一起）。硬编码在 Tool 里反而 _simple_（每个规则只有一个位置）。这也正是你 `docs/agent-design-principle.md` 里已经定下来的原则 —— 它不是偶然，它就是 Hickey 的这个建议在你项目里的一个具体实例。

## 3. 业界管理复杂度的通用思路（四个层级）

我把业界常用的手段按"作用层级"分成四档。每一档都是从不同角度去砍复杂度。

### 层级 1 — 代码层

| 思路                     | 大白话                                                                      |
| ------------------------ | --------------------------------------------------------------------------- |
| 小函数、单一职责、好名字 | 看起来平凡，但 legacy 代码的大部分痛苦都来自违反这几条                      |
| 类型即轻量证明           | TypeScript / Rust / 代数数据类型把一整类 bug 变成编译错误，尤其在领域建模时 |
| 纯函数和不可变数据       | 减少"状态可能给你惊喜"的面积。Hickey 的核心主张                             |

### 层级 2 — 模块层

| 思路                                                   | 大白话                                                                       |
| ------------------------------------------------------ | ---------------------------------------------------------------------------- |
| 信息隐藏（Parnas）                                     | 把"可能会变的东西"藏在接口后面                                               |
| 深模块（Ousterhout）                                   | 大功能、小接口                                                               |
| 依赖方向纪律（Clean / Hexagonal / Onion Architecture） | 领域逻辑不 import 基础设施；基础设施 import 领域。换数据库的时候领域层没感觉 |

### 层级 3 — 系统层

| 思路                                | 大白话                                                                                                                                                                                  |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Domain-Driven Design（Evans，2003） | 从领域专家实际使用的语言出发，让它驱动你的模型。把大系统切成 **bounded contexts**，每个 context 有自己的模型和自己的语言。跨 context 用翻译层。我见过最有效的一个对抗大系统复杂度的主意 |
| 分层 / 六边形架构                   | 规定依赖只能朝哪个方向指                                                                                                                                                                |
| CQRS 和 Event Sourcing              | 读模型和写模型分开；存事实，推导状态。当查询形状和更新形状差异很大时有用。对小项目是 overkill                                                                                           |

### 层级 4 — 流程和文档层

| 思路                                                  | 大白话                                                                                                                                                                                            |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Docs-as-code + ADRs                                   | 文档和代码放一起，进同一个仓库，走同一个 PR。**ADR (Architecture Decision Record)** 由 Michael Nygard 在 2011 年提出，用来捕获"_为什么_ 做这个决定"，追加写入，从不删除。三年后能回去读当时的推理 |
| C4 Model（Simon Brown）                               | 四层图：Context → Container → Component → Code。强迫你在不同缩放等级思考系统，避免所有东西糊成一坨。**你 `Single-Agent设计文档.md` 里已经在用 C4 L3 了**                                          |
| Living documentation / BDD / specification-by-example | 可执行的规格。规格和代码不一致时构建会挂                                                                                                                                                          |
| RFCs 和设计评审                                       | 任何非平凡的东西先写下来再动工。写的过程本身就会逼你发现漏洞                                                                                                                                      |

## 4. 对你这种"就一个小项目"的人意味着什么

以上这些文献里没有一条假设你一定是大团队。Ousterhout 的书是从教本科生出发写的。Brooks 写"No Silver Bullet"的时候也考虑过小项目。这些技术 **向下 scale 得非常好** —— 而且在你一个人的时候 **更有价值，不是更没价值**，因为：

1. 三个月后回来看代码的人就是你自己，没人能在走廊里给你讲背景。
2. 没有队友帮你在设计评审里发现烂抽象，烂抽象会直接固化下来。
3. 没有代码评审。

对独立开发者真正的风险 **不是流程太多**，而是 **选错了工具的 scale**。你不需要 Event Sourcing。你不需要微服务。但你 **确实需要**：

- **信息隐藏**（把会变的规则藏在模块里，而不是散在 prompt、tool、前端）
- **ADR**，把那些不那么明显的决定记下来（例如"为什么选手动 tool-call 循环而不用 AgentExecutor"、"为什么先做单 Agent 再加多 Agent"）
- **单一事实源（single source of truth）**，对那些反复绊倒你的领域规则（比如时间段规则、冲突检测规则）—— 每一条规则只能在代码里的一个位置、在文档里的一个位置出现

下一篇文档 `02-applied-to-chat-module.md` 会把这些具体落到你的 chat 模块上。
