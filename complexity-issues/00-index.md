## 每篇的一句话 TL;DR

### 01 · 软件工程复杂度到底指什么？
你的直觉对了。软件工程这门学科的一个干净描述就是"在需求变化、人员轮换的前提下长期管理系统复杂度"。Brooks / Parnas / Ousterhout / Hickey 这四位的经典观点给你一份可以直接落地的思想地图。

### 02 · 落到你的 chat 模块上
我读了 `packages/server/src/services/ai/` 的 9 个文件 ~1553 行。它没"写烂"，但同时扛着几种不同性质的复杂度，分 **三层**：领域复杂度 / 架构选择复杂度 / 实现偶然复杂度。按优先级最值得做的四件事是：**(1) 先写 ADR-0001 钉住"单 agent / 多 agent 主线"；(2) 把时间规则做成 table-driven tests 当作 AI 改代码时的边界锚点；(3) 给 AgentLoop 画状态图盘点退出路径；(4) 暂缓大规模 Rule Book / docs 重组，等主线和测试锚点都钉住后再做**。文末有几个我不确定的点，希望你回答一下，这样我能再收窄建议。

### 03 · 文档也是工程：一致性问题
不要追求"文档和代码同步"，追求"每个事实只住在一个地方"。PRD、设计文档、ADR、Rule Book、Runbook 是五种不同物种，不要混养。关于你的"代码是肉体、文档是精神"的比喻 —— 它抓到了一个真问题，但方向反了，**更有效的比喻是"地图与疆域"**：代码是疆域（真实发生的事），文档是地图（我们对它的表示）。

### 04 · AI 时代程序员的价值重心
AI 没有消灭复杂度，它只消灭了"打字"。结果是瓶颈从"写代码"往上游搬到了"讲清需求 + 定义正确 + 评审 AI 输出"。这三件事全都是复杂度管理能力。**独立开发者在 AI 时代比以前更需要懂软件工程理论**，不是更不需要 —— 因为你一个人能造的东西上限翻了 10 倍，复杂度也翻了 10 倍。

## 一条能贯穿所有 4 篇的主线

```
软件工程 = 管理复杂度
  ↓
好的复杂度管理 = 好的抽象 (simple, deep modules, 信息隐藏)
                + 好的文档 (意图、理由、放弃的方案)
                + 单一事实源 (规则 / ADR / 测试)
  ↓
在 AI 时代：
  这一整套能力 = 让 AI 好好干活的能力
  = 你作为独立开发者的核心竞争力
```
