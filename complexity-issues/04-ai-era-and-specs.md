# 04 · AI 编码时代：程序员的价值重心转移，以及它怎么绕回到复杂度管理（Q5）

> 回答你笔记里的问题：
> - 在 AI 辅助开发盛行的情况下，程序员是不是应该把更多时间和精力花在理解和掌控需求上？
> - 这和"管理复杂度"是不是同一个问题？

## 0. 先说结论

是的，两个问题都是对的，而且它们是同一个问题的两面。

- **AI 没有消灭软件复杂度，它只消灭了"打字"这一层**。复杂度的本质 —— 理解问题、定义正确、处理变化 —— 仍然完全在人身上。
- **程序员的价值重心正在往上游移**：从"写出可运行的代码" → "说清楚要做什么" + "设计验证" + "判断 AI 输出质量"。
- **这个上游能力就是复杂度管理能力**。你越懂得把复杂度切块、隔离、用规则表达出来，AI 就越有可能给你靠谱的代码；反过来，你越是"指着屏幕说帮我搞一下"，AI 就越会给你把偶然复杂度再叠一层。
- **一个正在升温的业界方向叫 Spec-Driven Development (SDD)** —— 把规格 (而不是代码) 当作真正的源头。它 **还没到"稳定的行业共识"** 的程度，但有真实工具、真实文献、真实收益报告，值得关注。不是什么新概念，但 AI 让它第一次有机会真正可行。

## 1. 一个正在升温的词：Spec-Driven Development (SDD)

过去 12–18 个月，业界围绕"AI 编码之后程序员该做什么"开始形成一个叫 **Spec-Driven Development** 的讨论方向。**需要先说清楚它现在的成熟度**：它还没有到"行业方法论已经稳定"的程度（一次 GPT 交叉审核在这里帮我收紧了语气 —— 2026 年 4 月的当下，它更准确地说是一个"明确升温中的工程方向"，而不是"已经沉淀的共识"）。但它是真实的 —— 有真实工具、真实文献、真实落地报告，不是假概念。值得关注，但不必立刻全面铺开。关键文献和工具：

- **GitHub Spec Kit**（github.com/github/spec-kit）：GitHub 官方开源的 SDD 工具链。2025–2026 期间 star 数涨得很快。提供 CLI 把"规格 → 实现计划 → 任务拆解 → AI 生成代码"这条链路串起来。
- **Thoughtworks**（2025 年的技术雷达 Blip）：把 SDD 列为"2025 年关键的 AI 辅助工程实践之一"。
- **arXiv 论文**《Spec-Driven Development: From Code to Contract in the Age of AI》(2026)：把 SDD 分成三层严格程度：spec-first（先写规格再写代码）、spec-anchored（规格锚定，但允许代码先行）、spec-as-source（规格就是唯一源，代码由它生成/验证）。
- 普遍说法："**意图是源头 (intent is the source of truth)**，代码是派生物。" 这个口号在过去 20 年出现过好几次（MDE、模型驱动开发、DSL），但都没成功 —— 因为代码生成器写不出真实世界代码。AI 让它第一次有机会跑起来。

**需要冷静的地方**：围绕 SDD 有很多 hype。真正落地的团队报告里你能看到很具体的收益（代码评审时间下降、新人上手时间缩短），但也能看到"规格写不好时 AI 就交付垃圾"这种老问题只是换了个位置。**这不是银弹。它只是把瓶颈搬家。** 对小项目来说，更合理的做法是 **先在高变化、高风险的模块局部试行**（例如你的时间规则和冲突规则），**从规则表 + 测试表开始**，而不是先搭一套大而全的 spec 流水线。

#### 展开：SDD 工具链具体在做什么？—— 以 GitHub Spec Kit 为例

SDD 听起来抽象，展开看它其实是一套 **分层模板 + 从模板到代码的流水线**。以 GitHub Spec Kit 为例，它给你 5 个层次，每一层是下一层的输入：

1. **Constitution** —— 项目长期原则、红线、不可妥协的约束。约等于你 `docs/agent-design-principle.md` 想做的事：工具要按用户意图封装、规则要在代码里不能在 prompt 里，等等。
2. **Specification** —— 某个功能要做什么（面向用户价值描述，不写实现）。约等于你 PRD 的某一节，但更机器可读。
3. **Plan** —— 技术方案、架构选择、分层。约等于你 `docs/Single-Agent设计文档.md`。
4. **Tasks** —— 任务拆解、依赖关系。约等于 Issue / TODO 列表。
5. **Implementation** —— AI 按前四层作为上下文生成代码。

**关键**：它 **不会自动生成** 这些文档，它只是给你一套 **结构化模板 + 分层约束**，逼你按顺序把每一层写出来，然后把每一层作为下一层的输入交给 AI。工具在解决的是"人容易跳步、容易只交一份模糊 prompt 就让 AI 动手"的问题，不是"人懒得思考"的问题。

回答你之前问的 "SDD 能帮我创建 PRD、规则文档等吗？"—— **能，但更准确地说是"它给你一套分层骨架，让你自己的思考按顺序填进去，不会再漏或者乱"**。内容还是你写的。

**对你当前项目的具体建议**：不建议立刻搬整套 Spec Kit —— 它的工作流偏重，还在演化中，你仓库的主线更需要先钉住（见 `02-applied-to-chat-module.md` §3 和 §5）。**但它的分层思路**（Constitution → Spec → Plan → Tasks → Implementation）**你可以立刻借鉴到 `docs/` 目录的整理上**。一句话：先偷它的目录结构，再考虑是否上它的工具链。

## 2. 瓶颈搬家之后，新瓶颈在哪？

AI 编码把"打字"的成本降到几乎为零。随之变化的不是"软件工程还需不需要"，而是"**软件工程里最花时间的那个环节换了位置**"。

**历史上的瓶颈**：

```
理解问题 → 设计 → 写代码 → 调试 → 改 → 部署
                  ^^^^^^^
                  这里最慢
```

**AI 之后的瓶颈**：

```
理解问题 → 设计 → [AI 打字] → 评审 AI 输出 → 验证 → 部署
^^^^^^^   ^^^^                ^^^^^^^^^^^   ^^^^
这里最慢 (模糊的理解会被 AI 无损地放大)
```

三件事现在变贵了：

### 2.1 把需求讲清楚

以前你可以"边写边想"。写了一段代码，发现 API 不好用，改掉；发现漏了一条分支，补上。思考和写作交织在一起，代码本身就在帮你澄清需求。

AI 帮你写代码之后，这个"写作促进思考"的循环变短了。你一句话过去，20 秒后一屏代码回来 —— 你还没来得及想清楚，实现已经存在了。**这时如果你需求描述不够精确，AI 会忠实地把你的模糊用确定的代码表达出来**。等你意识到这段代码干的是你没想过的事，它已经和另外三段代码耦合了。

结论：**提示词工程的真实含义不是"怎么哄 AI 干活"，而是"怎么把一个需求讲到让另一个智能体（哪怕是 AI）不需要猜就能实现"**。这和过去"给队友写一份能照做的任务说明"是完全一样的能力。

### 2.2 说清楚"正确"长什么样

当 AI 可以在 30 秒内给你 5 个实现版本时，你面对的新问题是：**哪一个是对的？**

"对"不能靠直觉。你需要可执行的"对的定义"。这就是 **测试** 的真正价值 —— 不是"为了防止回归"（那是副作用），而是"**它是你对'正确'的唯一机器可读的定义**"。

这也是为什么测试先行 (TDD) 这个老概念在 AI 时代被重新重视。在 AI 编码语境下它变成了：

1. 人写测试（或者描述场景让 AI 写测试，然后人审阅）。
2. AI 写实现。
3. 测试跑。
4. 不过 → AI 改实现，不是改测试。

**测试成了"规格"。** 它是人能写、人能审、机器能跑的那种规格。这比任何 Markdown 文档都更难作弊。

### 2.3 评审 AI 输出

你可能已经注意到了：AI 写代码的速度远远超过你能仔细读的速度。而且 AI 特别擅长写出"看起来对"但暗藏陷阱的代码 —— 变量名合理、结构合理、甚至注释都合理 —— 但里面有你没想到的副作用或者边界 case。

评审能力（知道该看哪里、知道什么叫"不对"）正在从"锦上添花"变成"核心技能"。这个能力不来自读 React 文档，它来自你对 **领域** 和对 **复杂度** 的理解深度。你越懂得"这个系统里哪些地方容易出 bug"，你评审 AI 输出时越能 30 秒定位问题。

## 3. 为什么这和"复杂度管理"是同一个问题

把上面三件事（讲清需求 / 定义正确 / 评审输出）连起来看，你会发现它们全都依赖同一件能力：**把一个领域的复杂度切块、命名、用规则表达出来**。

| 你在干什么 | 需要的能力 |
|---|---|
| 给 AI 写一个需求描述 | 把问题切成 AI 能一次处理的、边界清楚的块 → **信息隐藏** |
| 决定哪些规则写在 prompt 里、哪些写在 tool 里 | 判断"**easy** 的选择（塞给 LLM）和 **simple** 的选择（写到代码里）"的权衡 → **Hickey** |
| 给 AI 写测试让它照着实现 | 列出所有边界 case 和不变量 → **specification by example** |
| 读 AI 的输出找 bug | 知道这个领域"哪里容易藏问题" → **领域知识 + DDD** |
| 判断要不要接受 AI 的一个新抽象 | 判断"这个抽象是深模块还是浅模块" → **Ousterhout** |

这些全是 `01-what-is-complexity.md` 里那四位作者（Brooks / Parnas / Ousterhout / Hickey）已经讨论过的事。**AI 没有发明新问题，它让老问题变得更值钱了。**

**一个有趣的反转**：过去十年，独立开发者听到"软件工程理论"大多会觉得"那是大团队的事，我一个人不用管"。AI 让这个想法变危险了。因为：

- 过去，一个人能造的东西上限取决于"你一个人能写多少代码"。代码量小 → 复杂度自然小 → 你能 hold 住。
- 现在，一个人借助 AI 能造的东西上限翻了 10 倍。**代码量大了 10 倍 → 复杂度可能大了 10 倍 → 你一个人的理论功底反而更需要**。

独立开发者现在比以前更需要懂 Ousterhout，不是更不需要。你现在就是这个现象的活例子：你一个人就已经写出了一个 1500+ 行的 AI chat 模块，而且已经撞到了复杂度墙。

## 4. 对你的具体建议

### 4.1 思考方向上的调整

- **放弃"我再多写点代码就能掌控它"的想法**。写更多代码不是出路。**写更少的、单一事实源的、可被 AI 精确参考的规则**是出路。
- **开始把"需求描述能力"当成核心技能练**。每次给 AI 提需求时，问自己："一个完全不懂这个项目的工程师看到我这段话，能不能写出我想要的东西？" 如果不能，是我没讲清楚，不是 AI 不聪明。
- **把测试当成"规格"而不是"质量保险"**。写新功能时先写测试（或者先描述场景让 AI 写测试），然后再让 AI 去实现。这一步会把 90% 的模糊需求暴露出来。

### 4.2 对 `Home Task` 这个项目能立刻开始做的事

1. **把你 chat 模块里的"规则表"列出来**（时间规则、冲突规则、幻觉处理规则、工具选择规则）。哪怕只是 Markdown 表格。这一步会让你立刻看清"哪些规则彼此矛盾、哪些规则你已经忘了为什么"。
2. **挑一条最容易出 bug 的规则（例如"今天已过时段的追问"），写一张 table-driven 测试表**。每一行是一个 case，包括输入 + 期望行为。这张表就是这条规则的"规格"。
3. **下次给 AI 让它改 chat 模块时，把这张测试表一起放进 prompt**，让 AI 在规则下工作。观察 AI 的输出和你直接口头描述时有什么不同。
4. **为你现在已经撞到的复杂度墙写一条 ADR**，哪怕只有 10 行：标题类似 *"ADR-0001: AgentLoop 使用手动 tool-call 循环 + 跨轮状态追踪"*，把当时的理由和你现在的困惑都写下来。**这会迫使你把隐性知识显性化**，而这个过程本身就是复杂度管理。

### 4.3 长期的心态转变

以前你（和大多数独立开发者）的默认心态可能是：

> "我先写一版代码看看效果，不好就改。"

在 AI 时代，更有效的心态是：

> "我先把需求和测试写清楚，然后让 AI 把代码生出来。AI 的输出是我的草稿，不是我的定稿 —— 我对着规格和测试做评审。"

这个切换听起来像在多加一步，但实际上是 **把你的思考时间从 IDE 里移到 IDE 外**。大部分时候，思考的质量决定成品的质量，而 IDE 里的即时反馈反而会引诱你进入"一边写一边想"的糊涂状态。

## 5. 一个值得内化的框架

如果你只带走一个框架，带这个：

```
需求理解  →  规格 (可能部分可执行)  →  AI 写代码  →  评审 + 测试验证  →  合并

人花时间的地方：第 1 步 (最多) + 第 2 步 + 第 4 步
AI 花时间的地方：第 3 步
```

- **你花在第 1 步越多，第 4 步越少。** 反之亦然。
- **第 2 步是"第 1 步"和"第 3 步"之间的桥。** 没有桥，AI 就是在赌。
- **复杂度管理的能力决定了你第 2 步能做多好。** 所以复杂度管理 = AI 时代的核心竞争力。

这就是为什么"领悟需求"和"管理复杂度"在 AI 时代 **是同一件事**。你直觉是对的。
